# セキュリティポリシー - Jarvis CRM

## 脆弱性の報告

Jarvis CRMのセキュリティ脆弱性を発見された場合は、以下の手順でご報告ください。

### 報告方法

1. **非公開で報告**: セキュリティ問題はGitHub Issueで公開しないでください
2. **連絡先**: セキュリティチームに直接メールでご連絡ください
3. **報告内容**: 脆弱性の種類、再現手順、影響範囲を記載してください

### 報告に含めるべき情報

- 脆弱性の種類（XSS、SQLi、認証バイパス等）
- 再現手順（ステップバイステップ）
- 影響を受けるバージョン / エンドポイント
- 可能であればPoCコード
- 推奨される修正方法（任意）

### 対応プロセス

| ステップ | 期限 | 担当 |
|---------|------|------|
| 受領確認 | 24時間以内 | セキュリティチーム |
| 初期評価（重大度判定） | 48時間以内 | セキュリティチーム |
| 修正計画策定 | 5営業日以内 | 開発チーム |
| 修正リリース | 重大度による（下記参照） | 開発チーム |
| 報告者への完了通知 | 修正リリース後即時 | セキュリティチーム |

### 重大度と対応期限

| 重大度 | 定義 | 修正期限 |
|--------|------|---------|
| Critical | 認証バイパス、RCE、データ漏洩 | 24時間以内 |
| High | SQLi、権限昇格、IDOR | 72時間以内 |
| Medium | XSS、情報漏洩、CSRF | 2週間以内 |
| Low | 情報露出、設定ミス | 1ヶ月以内 |

## セキュリティ対策の概要

### 認証・認可
- Firebase Authentication + MFA
- JWT検証（RS256署名）
- ロールベースアクセス制御（admin/user）
- トークンブラックリスト（ログアウト対応）

### データ保護
- テナントスキーマ分離 + Row Level Security
- bcryptパスワードハッシュ（12ラウンド）
- TLS 1.2/1.3 通信暗号化
- HSTS有効（1年間）

### インフラセキュリティ
- 非rootコンテナ実行
- Dockerネットワーク分離（frontnet/backnet）
- Redis認証パスワード
- UFWファイアウォール（22/80/443のみ）
- Fail2Ban（SSH保護）

### 監視・ログ
- 全API認証イベントの自動記録
- CRUD操作の監査ログ（old/new値保存）
- Prometheus + Grafana メトリクス監視
- Loki ログ集約・異常検知
