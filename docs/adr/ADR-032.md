# ADR-032: Firebase Authentication カスタム認証ドメイン

## ステータス
採用済み（2026-05-27 本番適用完了）

## What
Sales Anchor の OAuth 認証フローで表示されるドメインを `sales-ops-with-claude.firebaseapp.com` から `auth.salesanchor.jp` に切り替える。

1. Firebase Hosting にカスタムドメイン `auth.salesanchor.jp` 追加
2. Firebase Authentication で authorized domains に追加
3. フロントエンド firebase config の authDomain を新ドメインに切り替え（環境変数 `VITE_FIREBASE_AUTH_DOMAIN`）
4. Meta Dashboard で新 Redirect URI を追加（既存 URI は保持）

## Why
- ユーザー（Meta App Review レビュアー含む）に見える OAuth URL が Sales Anchor ブランドで統一される
- Project ID 移行（9〜18 時間 + 高リスク）を回避しながらブランド統一を実現
- 撮影本番で `sales-ops-with-claude.firebaseapp.com` が映らない
- 将来の Project ID 移行時にも URL 変更不要

## Scope 外
- Project ID 自体の変更（撮影後の ADR-031 で対応）
- Firebase Hosting でアプリ本体をホスティングする（認証用途のみ）
- 他の Firebase プロダクト（Firestore, Storage 等）のドメイン変更
- DNS 管理権限の移行

## 事業上の制約
- 既存の `sales-ops-with-claude.firebaseapp.com` を併用維持（並行運用）
- 既存ユーザーへの影響を最小化（再ログインのみで対応）
- DNS 反映 + SSL 証明書発行に最大 48 時間
- Meta Dashboard で既存 Redirect URI を削除しない（撮影中の障害回避）
- 追加コストゼロ（Spark プラン無料枠で完結）
- 切り戻し容易（環境変数 `VITE_FIREBASE_AUTH_DOMAIN` を戻すだけ）

## 受け入れ基準
1. `auth.salesanchor.jp` で Firebase Authentication が動作
2. 新規ログインで OAuth URL が `auth.salesanchor.jp` 経由
3. 既存ログインセッションが壊れていない
4. 既存ユーザーが再ログイン後、正常動作
5. Meta OAuth が新ドメインで成功
6. 環境変数で旧ドメインへの切り戻しが可能
