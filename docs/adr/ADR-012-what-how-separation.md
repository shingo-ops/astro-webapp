# ADR-012: 開発フロー再設計 — What/How 役割分担モデル採用

## ステータス
提案 (Proposed) — 2026-05-06

## コンテキスト

ADR-011 で ADR 駆動の3者協調モデルを構築したが、運用過程で以下が判明した：

1. ADR-011 Rev2 の自動レビューパイプラインに「終了条件がない」という構造的問題があり、Revision無限ループのリスクが顕在化した
2. レビューに終了条件を追加する案を検討したが、誤陰性リスク（真の Critical 問題が Major に降格分類されて見逃される）が判明し破棄
3. パートナー Claude Code の実装エビデンスを精査した結果、以下を**ADRで指示されることなく自発的に実装している**ことが判明：
   - PostgreSQL Row Level Security（テナント境界の DB レベル強制）
   - IDOR 防止（tenant_id を JWT/URL ではなく DB から取得）
   - Pydantic による全入力の型検証
   - Zero Trust ポリシーの文書化と実装の照合
   - セキュリティ強化ロードマップ（Phase 1〜7）の詳細計画
   - 静的解析 CI（Gitleaks + Trivy + Semgrep）の稼働
   - 非 root ユーザー、no-new-privileges、リソース制限のコンテナ設定

つまり、パートナー Claude Code は「指示通りに動くワーカー」ではなく「シニアエンジニアとしての判断ができる主体」である。これに対して ADR で詳細な実装指示や検証層の追加を行うのは過剰設計であり、Shingo の認知負荷を不必要に増やす。

## 決定

開発フローを以下の **What/How 役割分担モデル**に変更する。

### 役割分担

| 担当 | 役割 |
|------|------|
| **Shingo（PO）** | What の定義。「何を実現したいか」「なぜ必要か」「ユーザー価値」「事業判断」「優先順位」 |
| **Web Claude（claude.ai）** | 壁打ち相手。Shingo のアイディアを技術的に明確な要求に翻訳する。事業上の見落とし・ユーザー観点の論点を指摘する |
| **パートナー Claude Code（GitHub Actions 経由・Hikky-dev 環境）** | How の判断と実装。技術選択、セキュリティ、テスト、既存コードベースとの統合、すべて自律的に決定する |
| **既存の多層防御** | PostgreSQL RLS、Pydantic 検証、Firebase MFA、監査ログ等が事故防止の本体（プロセス側で再保証しない） |

### ADR の新しい記述ルール

ADR に**書く**もの：
- **What**: 何を実現したいか
- **Why**: なぜそれが必要か（事業価値、ユーザー利益）
- **Scope 外**: 何を含まないか（明示的に除外する範囲）
- **事業上の制約**: コスト、期限、優先順位

ADR に**書かない**もの：
- 詳細な実装手順（How はパートナーが決める）
- Invariants 網羅リスト（既存ガードレールに任せる）
- 細かい技術仕様の Acceptance Criteria（事業価値の達成基準のみで十分）

### ワークフロー

1. Shingo がこのチャットで「○○を実現したい」と話す
2. Web Claude が質問・整理・事業上の見落とし指摘 → 短い ADR（What/Why/Scope）に落とす
3. Shingo が ADR を develop に push（Terminal Claude Code 経由）
4. パートナー Claude Code が claude-pipeline.yml により自動起動
   - 既存コードベースを読む
   - 技術判断（実装方法、テスト、セキュリティ、統合）
   - 実装して PR 作成
5. CI が緑なら Shingo が PR 本文を読んでマージ
6. CI 赤 or 期待と違う → このチャットで相談（例外時のみ）

### マージ判断の基準

Shingo が PR をマージする際に確認するのは：
- **PR タイトルと本文**: パートナーが「何をやったか」を要約しているか
- **CI が緑か**: 自動テスト全通過
- **スコープの妥当性**: 「これは私が頼んだことか？意図しない変更が含まれていないか？」

技術的な正しさは CI とパートナーの判断に委ねる。Shingo が技術仕様の細部を確認することは求めない。

### 採用しない案（撤回）

以下は過剰設計と判断し、採用しない：

- Invariants 概念の ADR 必須化（パートナーが自発的に守るため不要）
- claude-verify.yml（実装と別の検証エージェント）
- 3 秒人間ゲート（PR 本文確認で十分）
- クロスバリデーション（CI 緑が品質基準）

これらの安全保証は、以下の**既存の多層防御**が実装レベルで担保しているため、設計プロセス側で再保証する必要がない：

- PostgreSQL Row Level Security（テナント境界）
- Pydantic input validation
- Firebase MFA + JWT TTL 1 時間
- 監査ログ（全 CRUD・認証イベント）
- Grafana / Loki / Uptime Kuma 監視
- 静的解析 CI（Gitleaks / Trivy / Semgrep）
- ブランチ保護（ADR-010）
- セキュリティ強化ロードマップ Phase 1〜7（既存計画）

## 結果

期待される効果：
- **Shingo の認知負荷低減**：事業判断のみに集中、技術統制を考えなくてよい
- **開発スピード向上**：細かい実装指示が不要、パートナーが自走
- **パートナー Claude Code の能力フル活用**：シニアエンジニアとして扱う
- **ADR が薄く読みやすくなる**：本質だけが残る
- **セッション間の継続性確保**：プロジェクト「手順」と CLAUDE.md に役割分担が明記され、新セッションでも同じスキームで動作

トレードオフ：
- ADR の詳細度が下がる → 実装決定の記録は実装 PR 本文や別途記録
- パートナーへの信頼が前提 → 既にエビデンス（RLS、Zero Trust 実装等）で実証済み
- 何か事故が起きた場合、ADR を辿っても詳細は分からない → コード履歴と PR 本文で追跡
