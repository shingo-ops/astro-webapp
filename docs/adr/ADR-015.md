## Context

### 現行CS運用の課題

現在、新規リードへの初期対応はCSが人力で行っている。

```
顧客が問い合わせ
  ↓ CSが対応（2〜3回のやり取り）
Q1: どの国ですか？
Q2: 何のタイトルに興味がありますか？
  ↓ 返信速度をメモ
  ↓ Shingoにアサイン連絡
```

問題：
- CSリソースを単純なルーティング業務に使っている
- 返信速度の記録が属人的でムラがある
- 問い合わせチャンネルが複数（Messenger / Instagram / Discord / Telegram）あり、同一人物の重複対応が発生している
- 顧客情報がGASスプレッドシートに散在し、営業の引き継ぎロスがある

### エビデンス

本ADRは以下の調査・実績に基づく：

- GAS WebApp カルテ現行運用の調査（2026-05-07 パートナー実施）
- CRM/SFA導入事例 成功5件・失敗5件の分析
  - **失敗の最大原因**：営業担当の入力負担増による「使われないCRM」
  - **成功のパターン**：Perimeter Real Estate が「次のアクションを設定するだけ」のルールで50%事業拡大
- 実際の商談ログ（Kasper・Paolo）から抽出したカルテ項目の有効性確認
- Klarna失敗事例：AI全面代替による顧客満足度低下

## Decision

### §1 AIがやること・やらないこと（境界線）

**CSと完全に同じことだけをAIにやらせる。それ以上はやらない。**

AIが自動でやること：
- 新規リードからQ1（国）・Q2（タイトル）を収集する
- 返信速度を自動記録する
- 情報が揃ったらShingoにDiscordで即通知する
- 会話・売上DBからカルテ情報を自動抽出・候補提示する（「商談を解析」ボタン）

営業担当（人間）だけがやること：
- 温度感の判断
- 商談の手応え・所感メモ
- 次回アクションの内容と日付の設定
- 成約・失注・アーカイブの判断

**理由：Klarnaの失敗事例より。AIが営業判断に介在すると顧客関係が損なわれる。**

### §2 情報収集の最小化原則

収集するのは2項目のみ：

| 質問 | 内容 | 必須 |
|---|---|---|
| Q1 | 配送先の国 | ✅ |
| Q2 | 興味のあるタイトル（Pokemon / One Piece 等） | ✅ |
| 自動記録 | 返信速度（問い合わせ〜返信の時間） | 自動 |

数量・予算・購入時期は営業担当が直接ヒアリングする。AIは収集しない。

### §3 新規リードのフロー

```
顧客が問い合わせ（どのチャンネルからでも）
  ↓ SNS IDで顧客検索
  ├─ 既存顧客 → AIは対応せず、担当営業に即通知
  └─ 新規    → リード作成 → AIがQ1→Q2を収集

AIのやり取り（最大2〜3回）：
  1回目受信 → 「Which country are you shipping to?」
  国を受信  → 「Which title are you interested in?」
  タイトル受信 → 返信速度を記録 → 即アサイン → Shingoに Discord通知
             → 「Thank you! Our sales team will reach out to you shortly.」
```

既存顧客への対応：
- SNS IDで既存顧客と判定した場合、AIは一切対応しない
- 担当営業に即通知して直接対応させる
- **理由：既存顧客は関係が構築済み。AIが割り込むと不自然になる**

対応できないケースの処理：
- 理解できないメッセージ：最大1回聞き返す → それでも取れなければ「要人間確認」フラグ → Shingoに通知
- スパム・無関係メッセージ：自動アーカイブ候補として Shingo が確認・判断

### §4 カルテのAI補助設計

カルテ項目とAI介入レベル（確定仕様）：

| カルテ項目 | 分類 | 備考 |
|---|---|---|
| 国 | 自動 | 会話から直接抽出 |
| 取り扱いタイトル | 自動 | 会話から抽出 |
| 返信速度 | 自動 | システム計算（AI不要） |
| 顧客タイプ | AI提案→人間確認 | 実店舗/オンライン/転売 |
| 販売形態 | AI提案→人間確認 | 会話から判断して候補提示 |
| 競合比較中 | AI提案→人間確認 | 「他社とも話している」等を検知 |
| CSメモ | 自動（人間補足可） | AIが下書き、人間が加筆可 |
| 想定規模 | 自動 | テナントが設定した金額基準で分類 |
| 1回の発注金額 | 自動 | 新規：見積もりから / 既存：売上DBから |
| 購入頻度（月次） | 自動 | 新規：会話から / 既存：売上DB実績から |
| 月間見込み金額 | 自動 | 上記から算出 |
| 相手の課題 | 人間（将来AI補助検討） | 現在は営業担当が記入 |
| 温度感 | 人間のみ | AIは介在しない |
| 呼び方（英語） | 人間のみ | AIは介在しない |
| 商談の手応え | 人間のみ | AIは介在しない |
| 商談メモ | 人間のみ | AIは介在しない |
| 次回アクション | 人間のみ | 設定後にダッシュボードと連動 |
| 次回アクション日 | 人間のみ | ダッシュボードのリマインドに連動 |

AI補助のトリガー：「商談を解析」ボタンを押した時に全フィールドを一括処理する。
処理結果はすべて「候補として薄く表示」し、営業が確認・承認してから保存する。

使用するLLM：
- 単純な情報抽出（国・タイトル）：ASTRO FORCE Server BのQwen3 8B（自前サーバー、コスト¥0）
- 複雑な分析（顧客タイプ・販売形態・競合検知・CSメモ）：Claude Haiku 4.5 API
- 既存顧客の定量情報（発注金額・頻度・月間見込み）：売上DBの直接計算（AI不要）

### §5 ダッシュボード設計

**設計思想：ログインした瞬間に「今日やるべきこと」が全部見える**

エビデンス：Perimeter Real Estate が「次のアクションを設定するだけ」のルールで50%事業拡大。OnePageCRM の Next Action 方式が成功の核心。

リマインドの分類：

| セクション | 条件 | 優先度 |
|---|---|---|
| ⚠️ OVERDUE | 期日を過ぎている | 最高・最上部に固定・赤 |
| 🔴 TODAY | 本日が期日 | 高・赤 |
| 🟡 TOMORROW | 明日が期日 | 中・黄 |
| 🟢 UPCOMING | 3日以内が期日 | 低・緑 |

画面下部に今月のサマリー（新規リード数・商談中・成約・AI対応中・アーカイブ）を表示する。

担当者フィルタリング：
- デフォルトは「自分の担当案件のみ」表示
- 管理者（Shingo）は全担当者の案件を横断して確認可能

### §6 ステータス設計（確定版）

```
新規（AI対応中）
  ↓ 国 + タイトルが揃ったら
商談中
  ↓ 成約
既存顧客（ルート営業対象）
  ↓ 何らかの理由でクローズ
アーカイブ
  ├─ 追客（短期）：3ヶ月以内に再アプローチ → 30日後にリマインド
  ├─ 追客（長期）：3ヶ月以上先 → 90日後にリマインド
  ├─ 失注
  └─ 対象外
```

### §7 テナント設定（プレイブック）

Metaの固定テンプレートと異なり、テナントがAI対応フローを自分でカスタマイズできる。

テナントが設定できる項目：
- 挨拶メッセージ
- 収集する情報リスト（質問テキスト・必須/任意・順番）
- アサイン条件（全必須項目収集後 / 最初の返信後 / N回やり取り後）
- アサイン後メッセージ
- 担当者割り当て方法（手動 / ラウンドロビン / 国別固定）

---

## Scope 外

- 数量・予算・購入時期のAI収集（営業が直接ヒアリングする）
- 商談中のサブステータス（今回スコープ外。現行実装の確認をパートナーに委ねる）
- 受注管理モジュール連動（別ADR）
- AI全面自動応答（人間の判断領域は侵さない）

---

## 事業上の制約

- **ASTRO FORCE Server B（Sakura VPS 16GB / ¥8,800/月）が自前LLMサーバー**。単純な抽出タスクはここのQwen3 8Bで処理し、Claude APIコストを最小化する
- AI補助コストの目安：約¥0.4〜1.4/商談、月150商談で¥60〜210/月
- 既存のDiscordアサイン通知チャンネルに統合する（新規チャンネル不要）
- マルチテナント構成（`{schema}.` プレフィックスパターン）に従う

## パートナーClaude Codeへの委任事項

以下の技術判断はすべてパートナーに委ねる：

1. DB設計（テーブル・カラム・インデックス・マイグレーション番号）
2. Qwen3 8B（ASTRO FORCE Server B）とClaude Haiku 4.5の呼び出し実装
3. 既存コードとの統合方法（`analyzeDeal`等の既存実装との整合性）
4. `product_master`（商品マスタ）との連携方法
5. Celery Beatバッチのスケジュール設計
6. フロントエンドのコンポーネント設計
7. 商談中のサブステータスが既存実装に含まれているかの確認と対応判断
