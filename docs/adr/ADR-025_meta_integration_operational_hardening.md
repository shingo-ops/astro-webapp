# ADR-025: Meta 連携の運用整備強化 - 環境変数注入の確実性とパートナー実装ガイドラインの強化

- **Status**: Proposed
- **Date**: 2026-05-12
- **Deciders**: Shingo Tanizawa (PO)
- **関連 ADR**: ADR-012（What/How 役割分担）, ADR-024（Meta 連携の構造的不整合の修正）

---

## Context

ADR-024 にて Meta 連携の構造的不整合（subscribed_apps 未呼出、暗号化キー不一致、検証機構の欠如）を修正したが、ADR-024 の Scope 内であるべき **deploy.yml の環境変数注入パターンの構造的欠陥**が PR #335 で対応されなかった。

具体的には deploy.yml が以下のパターンを使用している：

```bash
grep -q "^METADATA_FERNET_KEY=" .env 2>/dev/null || \
  echo "METADATA_FERNET_KEY=${{ secrets.METADATA_FERNET_KEY }}" >> .env
```

このパターンは **「.env に既存行があれば上書きしない」設計**で、GitHub Secrets を更新しても VPS の .env に反映されない。これは 2026-05-01 の鍵更新が VPS に反映されず、毎日の token refresh 失敗の直接原因となった。

加えて、ADR-024 起案の経緯から以下の運用上の盲点も判明している：

- **データの手動 DB INSERT を許す運用**が「DB上は接続済み・Meta上は未接続」の不整合を生んだ
- **Phase 完了判定に実機 E2E テストが含まれていなかった**ことで、Meta webhook 受信不可の状態を Phase 完了として承認してしまった
- **検証・監視・運用面の項目が ADR の Scope に含まれない場合、パートナー Claude Code が実装しない**設計判断パターンが明らかになった

これらは Phase 1（App Review 通過）の直接的な阻害要因であり、本番運用後の障害再発防止にも必須の改善である。

## What

以下の状態を実現する：

1. **環境変数注入の確実性**: GitHub Secrets を更新したら必ず VPS の .env に反映される deploy.yml の構造に修正する
2. **暗号化キーローテーション手順の整備**: METADATA_FERNET_KEY を含む暗号化キーの更新時に、データ整合性を保つための運用手順書を整備する
3. **データ手動投入の原則禁止**: OAuth フロー外でのデータ DB 投入を運用ガイドラインで原則禁止し、例外条件を明確化する
4. **Phase 完了判定基準の強化**: 外部 API 連携機能を含む Phase / Sprint の完了判定に「実機 E2E テスト」を必須化する
5. **パートナー実装ガイドラインの強化**: 新機能実装時に「機能本体 + 検証スクリプト + 監視・アラート」を 3 点セットとして実装することを CLAUDE.md に明文化する

## Why

- **直接的事業要因**: ADR-024 で修正した内容を確実に本番反映するには、deploy.yml の構造的欠陥を先に解消する必要がある。これなしでは PR #335 のマージとデプロイを行っても、再接続後の運用で同じ問題が再発する
- **本番運用上の致命的問題の再発防止**: 環境変数の不整合は Meta 連携以外（決済 API、外部メール、認証基盤など）でも同様の障害を引き起こす可能性が高い
- **検収プロセスの再現性**: 今回 Meta App Review 撮影直前で発覚した「Phase 完了判定の甘さ」を放置すると、本番テナント追加・新機能リリース時に類似の事故が再発する
- **パートナー Claude Code の実装品質の底上げ**: ADR 起案時に検証・監視を Scope に含めなかった場合、実装されない設計パターンが今回判明。今後の Phase / Sprint で同じ漏れを起こさないために CLAUDE.md レベルでのガイドライン化が必要
- **データ整合性の構造的保証**: 「DB上は接続済み・Meta上は未接続」のような不整合状態を、運用ルールと検証スクリプトの両面から防止する必要がある

## Scope 外

- ADR-024 で実装済の検証スクリプト本体（verify_meta_subscriptions.py）の機能変更
- 新規機能の追加（受信ロジック、UI 変更、新 Permission の申請等）
- 暗号化アルゴリズム自体の変更（Fernet 継続）
- ADR-023（staff lifecycle 3 層同期）の実装範囲
- Phase 2 で扱う「検証スクリプトを Meta 以外（決済・認証等）にも拡張」の作業
- Sentry / Slack 通知の新規導入（既存通知機構があれば連携、なければ Phase 2）
- migration を伴うスキーマ変更
- 既存の正常な接続レコードへの影響を伴う処理

## 事業上の制約

- **Meta App Review 撮影の前提条件**: 本 ADR の実装完了まで撮影は再開しない
- **最優先**: deploy.yml の構造修正は緊急性最高。これが完了しない限り PR #335 のデプロイが意味を持たない
- **正規ルート遵守**: VPS 直接作業は最小限とし、可能な限り deploy.yml 経由で本番反映する
- **既存運用への影響最小化**: deploy.yml の変更は段階的・冪等的に行い、デプロイ失敗時のロールバック手順を明示する
- **CLAUDE.md / docs 改修**: パートナー Claude Code 自身が読むファイルの変更が含まれるため、変更後の整合性確認が必要

## 受け入れ条件（実装完了の判定基準）

### deploy.yml 修正

1. deploy.yml で全ての GitHub Secrets 由来の環境変数注入が「Secret 更新時に必ず VPS .env に反映される」パターンになっている
2. 修正後の deploy.yml で 1 回デプロイし、VPS .env の METADATA_FERNET_KEY が GitHub Secrets の現在値と一致することを目視確認できる
3. 既存の正常な環境変数注入が引き続き動作する（破壊的変更がない）

### 運用ガイドラインの整備

4. `docs/operations/meta_encryption_key_rotation.md` を新規作成し、暗号化キーローテーション手順（接続済テナントへの影響、再接続フロー、検証コマンド）を記載
5. `CLAUDE.md` に「データ手動 DB INSERT の原則禁止」セクションを追加（例外条件、audit_log 記録義務、検証スクリプト併用を明記）
6. `CLAUDE.md` に「Phase / Sprint 完了判定基準」セクションを追加（実機 E2E テスト必須、検証スクリプト並行実装、監視・アラート機構整備の 3 点セット）
7. `CLAUDE.md` に「新機能実装時の 3 点セット要件」セクションを追加（機能本体 + 状態検証スクリプト + 監視・通知）

### 検証

8. 本 ADR 実装後のデプロイで、`meta_token_refresh_failed` audit_log が翌日 18:00 JST に新規発生しないことを確認できる状態である（ただし METADATA_FERNET_KEY の整合化と再接続が前提）
9. Meta App Review 撮影台本 Scene 5（OAuth 接続）が問題なく実演できる

## 関連する Phase 2 への持ち越し項目

本 ADR の Scope 外だが、撮影通過後の Phase 2 で必ず対応すべき項目：

- 監視・アラート機構の本格整備（Sentry または Slack/Discord 通知）
- 検証スクリプトの拡張（Meta 以外の外部連携への適用）
- 接続レコード手動操作の audit_log 記録自動化
- 環境変数管理の構造改善（HashiCorp Vault 等の検討）

これらはバックログに記録し、Phase 2 着手時に優先順位を整理する。

## 補足: ADR-024 との関係

本 ADR は ADR-024 の **補完** であり、ADR-024 で見落とされた Scope を明示的に補う位置づけとする。本来は ADR-024 にこれらを含めるべきだったが、パートナー Claude Code が deploy.yml 修正と運用ガイドライン整備を Scope 外と判断したため、独立した ADR で再依頼する。

この経験は ADR 起案者（Shingo / Web Claude）への教訓でもあり、今後の ADR 起案時に「コードレベルの修正だけでなく、運用・監視・検収プロセスまで Scope に含めるか」を明示的に判断する。
