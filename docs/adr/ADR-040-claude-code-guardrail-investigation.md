# ADR-040: Claude Code 運用ガードレールの存在性調査

## ステータス
Draft - 調査フェーズ

## 背景（Context）

Terminal Claude Code が ADR-037 の起案フローで連続2回のワークフロー違反を起こした：

1. **1回目**: VPS（ubuntu@49.212.137.46）に直接SSHして git 作業を試行。
   プロジェクト指示の「VPS への直接SSH接続は禁止」に違反。

2. **2回目**: develop ではなく main をベースに feature ブランチを作成し、
   ADR-037 起案と無関係な19コミットを含むPR #382を main 向けに作成。
   プロジェクト指示の「develop に push」フローから逸脱。

Web Claude（チャット）からの指示は明確で、禁止事項も明示していたにもかかわらず
違反が発生した。これは推測ではなくエビデンスで原因を特定する必要がある。

「ガードレールが存在しない」のか「存在するが機能していない」のか、
あるいは「指示の伝達経路に欠陥がある」のか、現時点では確定していない。

## What（何を調査するか）

以下の事実を確認し、調査レポート（Markdown）をPRで提出する。
本ADRはあくまで調査依頼であり、ガードレール追加の実装は別ADR。

### 調査項目1：既存ガードレールの存在確認

- CLAUDE.md に以下の記述があるか、ある場合の具体的な文言：
  - ADR起案フロー（develop から切る / ファイル1個のみcommit / develop に向けたPR）
  - VPS 操作の禁止規定
  - feature ブランチの命名・運用ルール
- .github/workflows/ に以下の自動チェックがあるか：
  - PRのベースブランチ妥当性検証
  - PR内コミット数の異常検知
  - ADR起案PR専用のlint
- ADR-012 で定義したフローと、現在のCLAUDE.md / workflow定義との一致度

### 調査項目2：Terminal Claude Code の起動・実行環境

- Terminal Claude Code がどのディレクトリ・どのマシン（Mac/VPS）で起動するかの規定
- セッション開始時に CLAUDE.md を自動読み込みする設定の有無
- 「現在の作業環境が Mac か VPS か」を自己確認する仕組みの有無

### 調査項目3：直近2回の違反の根本原因

- 1回目（VPS SSH）：なぜVPSを選んだのか。Macクローンが存在しないからか、設定上の問題か
- 2回目（main ベース PR）：なぜ develop ではなく main から切ったのか
- 19コミット混入：feature ブランチを切るタイミングで develop が pull されていなかったか、
  そもそも feature ブランチを古い main から切ったか

### 調査項目4：所見

- ガードレールが「存在しない」「存在するが機能していない」「指示伝達経路の欠陥」
  のどれに該当するかの判定
- 各原因に対して防げる追加対策（CLAUDE.md改訂 / GitHub Actions追加 / 起動スクリプト整備 等）
- 推奨される次のADRの方向性

## Why（なぜこの調査が必要か）

- 撮影中の不安定な時期に2回連続のワークフロー違反は許容できない頻度
- 推測ベースで「ガードレールがない」と仮定して対策ADRを起案すると、
  根本原因が別にあった場合に対策が的外れになる
- ADR-012 で定義した「Claude Code を信頼する」原則を維持しつつ、
  信頼の範囲を明示するには、現状の信頼境界を事実確認する必要がある

## Scope外

- 対策の実装（本ADRは調査のみ）
- ADR-012 そのものの改訂
- 全ADR レビュー
- Terminal Claude Code の根本的な再設計

## 出力形式

- `docs/research/` 配下に調査レポートを配置
- 「現状の事実（コード/設定から確認できたこと）」と「Claude Codeの所見」を明確に分離
- 各違反パターンに対する根本原因の特定と、対策の選択肢を列挙

## 事業上の制約

- 撮影が止まっているため、本調査は短時間で完結させる
- 完璧な分析より、次のADR起案・ADR-037 やり直しに必要な最低限の事実確認を優先
- 本ADRの調査完了まで、Terminal Claude Code に新規作業を依頼しない
