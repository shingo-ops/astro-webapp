# ADR-001: ADR駆動ワークフロー採用 (3者協調モデル)

## ステータス
提案

## コンテキスト

salesanchorは以下の3者で開発される：

- **Shingo (PO)**: 事業判断と要件定義を担うが、コードレベルの技術判定はできない
- **Web Claude**: 設計・アーキテクチャ判断を担う相談相手。リポジトリ実体には触れない
- **パートナーのClaude Code**: セルフホストランナー上で実装・コードレビューを担う

このとき以下の課題が存在する：

1. 設計者(Web Claude)はコード実体を見ずに設計するため、現実乖離リスクがある
2. 実装者(Claude Code)は設計意図の解釈に迷っても、誰に質問すべきか経路がない
3. ShingoがブリッジになるがShingoは技術判定できないため、認識ズレが沈黙のまま蓄積する
4. ADR文書1種類だけでは「判断の根拠」と「実装指示」が混在し、Claude Codeの実装精度が落ちる
5. 現在のレビュー起動は手動のため、壁打ちサイクルが遅い

## 決定

### 1. ドキュメント2層構造

- **ADR (`docs/adr/ADR-XXX.md`)**: 判断の根拠を残す。「なぜそうしたか」のみ記述。
- **Spec (`docs/spec/SPEC-XXX.md`)**: Claude Codeへの実装指示。受入基準は非技術者が判定可能な粒度で書く（例: 「ログイン後3秒以内にダッシュボードが表示される」）。
- ADRとSpecは相互に参照IDを持つ。

### 2. レビューワークフローの自動トリガー化

`.github/workflows/claude-max-auto-pipeline.yml` のトリガーを以下に変更する：

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'docs/adr/**.md'
      - 'docs/spec/**.md'
  workflow_dispatch:  # 手動も併用可
    inputs:
      adr_files: ...
```

### 3. フィードバック復路の確立

- パートナーのClaude Codeはレビュー結果を `feedback.md` に追記する（既存運用）。
- ShingoはWeb Claude (claude.ai) のSettings > ConnectorsでGitHubコネクタを有効化し、本リポジトリへの読み取り権限を付与する。
- これによりWeb Claudeは `feedback.md`、既存ADR、既存Specを直接参照可能になり、設計時の現実乖離を抑制する。

### 4. 質問返しチャネル

- Claude Codeが設計の解釈に迷った場合、**独自判断で実装を進めず**、`questions/QXX.md` に質問を書いて作業を停止する。
- ShingoはそれをWeb Claudeに共有し、回答を対象ADR/SpecへのAmendment commitとして反映する。

### 5. 実装差分レポートの義務化

- 実装完了時、Claude Codeは Spec ↔ 実装の差分を `docs/diff-reports/DIFF-XXX.md` に記録する。
- 記述項目: 「Spec通り完全実装の項目」「妥協した項目とその理由」「未実装項目」。
- 差分があればShingoがWeb Claudeに共有し、必要に応じてADR追記またはSupersede。

### 6. ADR/Spec起案の単一主担当

- ADR/Specの起案とpushは **ShingoのターミナルClaude Code経由に一元化**する。
- パートナー向けに用意されているWeb Claude Project「ADR作成」は、Shingo不在時のフォールバック用途に格下げする（連番衝突回避のため平時は使用しない）。

### 7. push経路とブランチ運用

- ShingoのターミナルClaude Codeから `main` 直pushを許容する。
- ADR/Specはコードと異なりビルド失敗が起きないため、PR運用のオーバーヘッドに対し効果が薄いと判断。
- 将来パートナーが複数になった場合は再検討する。

## 結果

**得られるもの**

- 設計→実装の認識ズレを検出する機構が成立する
- Web Claudeとパートナー Claude Code の壁打ちサイクルが分単位で回る
- 非技術者であるShingoは「コピペとAcceptを判断する係」に集中でき、技術的優劣の判定が不要になる
- Web Claudeが既存ADR/Specを参照できるようになり、設計が現実から乖離するリスクが下がる

**トレードオフ**

- パートナーへの依頼が3件発生する（ワークフローYAML修正、`feedback.md` 追記運用継続、`questions/QXX.md` の解釈合意）
- ShingoはGitHubコネクタの有効化作業が必要
- main直push継続のため、不適切なADRが一時的にmainに乗るリスクがある（Supersedingで対応可能）
- パートナーのClaude Code (Hitoshiさんの Mac) の常時稼働要求が高まる

**本ADR施行のために必要なアクション**

1. ShingoがClaude.aiのSettings > ConnectorsでGitHubコネクタを有効化、本チャットでON
2. パートナーに `claude-max-auto-pipeline.yml` のトリガー変更を依頼・合意
3. Shingoローカルに `shingo-ops/salesanchor` をclone
4. `docs/spec/`、`docs/diff-reports/`、`questions/` ディレクトリを作成（空なら `.gitkeep` を置く）

**Supersedeルール**

本ADRの個別決定（1〜7）はそれぞれ独立した別ADRで上書き可能。上書き時は本ADRの該当章番号を明記すること。
