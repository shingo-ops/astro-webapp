# ADR-014: 在庫管理モジュール — Discord 自動収集 + AI 解析 + 自己改善ループ

## ステータス
提案 (Proposed) — 2026-05-07

## コンテキスト

### 現状の課題

Treasure Island JP / HIGH LIFE JPN は、TCG（トレーディングカードゲーム）の B2B 輸出事業者として、複数の国内仕入元から日々在庫情報を受け取っている。現行の運用は以下の通り：

- 仕入元は各社の任意フォーマット（自由文・画像・スクショ混在）で Discord チャンネルに在庫を投稿する
- Shingo が GAS（Google Apps Script）+ スプレッドシートで在庫情報を構造化・集計する
- スプレッドシート上の `GPT_Knowledge` シートを商品マスタとして、自由文を AI で正規化する
- クライアント（バイヤー）には、整形済みの在庫一覧をスプレッドシート経由で共有する

この運用には以下の限界がある：

1. **GAS の処理が完全一致前提で、表記ゆれに弱い**：仕入元の表記が少しでも異なると未知商品扱いになり、毎回手動で修正する必要がある
2. **解析精度が運用しても上がらない**：修正履歴がスプレッドシートに散在し、AI に学習データとして再利用されていない
3. **仕入元固有のクセ（特定の略語・単位・販売形態）を仕入元別に管理しにくい**
4. **Sales Anchor と分離されている**：CRM 側の商品マスタ（`products`）・受発注・見積と連動しておらず、二重管理が発生
5. **解析ロジック・プロンプト・仕入元の存在自体がスプレッドシートに露出している**：クライアントに見せるシートと管理用シートが同じファイル内で運用されており、情報秘匿が脆い
6. **価格・数量の推移が時系列で追えない**：在庫が「いまいくらか」しか見えず、相場感の把握ができない

### なぜ Sales Anchor 内で実装するのか（事業価値）

- **運用するほど精度が自動で上がる仕組み**：人手の修正コストが時間とともに逓減する
- **解析ロジック・プロンプト・仕入元情報の秘匿性確保**：差別化の源泉である「どの仕入元から、どのプロンプトで、どう正規化しているか」をクライアントから構造的に見えなくする
- **既存 CRM 機能（顧客・見積・請求・注文）との地続き化**：在庫を起点に営業・受注業務が一気通貫で動く土台を作る
- **GAS + スプレッドシート運用からの完全離脱**：属人化・スクリプト破損リスク・スケール限界の解消

## 決定 — What を実現するか

Sales Anchor 内に在庫管理モジュールを実装する。実現したい価値は以下のとおり。

### 1. Discord 仕入元チャンネルの自動収集

Bot が在庫専用チャンネルのメッセージ（テキスト＋添付画像）を自動で取り込み、原文を保全したまま解析パイプラインに流す。CSV / Excel アップロードのような能動操作は不要。

### 2. AI による構造化解析

仕入元の自由文メッセージを「商品名・状態・数量・価格・売り切れ判定」等の構造化データに変換する。

- **コスト抑制を最優先で Gemini Flash Lite を一次採用**する。本番運用しながら品質を実測し、目標水準に届かない場合は上位モデル（Claude Haiku 系・他社上位モデル）への切り替えを検討する。閾値・コスト上限・切り替え判断基準は運用開始後に PDCA で決める
- 解析結果には信頼度スコアを付与する

### 3. 信頼度ベースの承認フロー

- 高信頼度 → 自動承認して在庫マスタに反映
- 低信頼度・未知商品 → Shingo の Discord に詳細通知が届き、手動レビューする

### 4. 修正履歴の蓄積と自己改善ループ

人間（Shingo）が低信頼度結果を修正した履歴を「学習データ」として蓄積する。蓄積された履歴を **Claude（Web Claude または Claude Code）が定期的に分析し、解析品質を改善する仕組み**を持つ。これは Gemini Flash Lite 単体では実現できない高次の推論を要するため、別レイヤーとして設ける。

改善ループの主体（手動壁打ちか定期バッチか両方か）・トリガ条件・改善内容の反映経路は、運用しながらパートナーが設計する。

### 5. 仕入元別の精度監視

仕入元ごとに解析精度を継続計測し、精度急落時は通知、精度安定時は自動承認の度合いを引き上げる等の運用判断材料を提供する。

### 6. 価格・数量の時系列可視化（**管理者のみ閲覧**）

承認済み在庫の価格・数量推移を、商品×状態別に時系列でチャート表示する。相場観の把握とクライアントへの提案根拠として使う。**クライアント（バイヤー）には見せない**。

### 7. クライアント別の整形済み在庫一覧

クライアントごとに表示条件（価格上限・優先仕入元・並び順・重複排除ルール等）を設定でき、その条件で整形した在庫一覧のみをクライアントに提示する。**解析ロジック・プロンプト・仕入元の存在・修正履歴・チャート・精度ログはクライアントから完全に不可視**にする。

### 8. GAS からの初期データ移行

既存 GAS の `GPT_Knowledge` シート（商品マスタ）と仕入元別ルールを Sales Anchor に移行する。

## Scope 外

以下は本 ADR では扱わない：

- **CSV / Excel アップロード機能**：Discord 自動収集で代替するため不要
- **TCGdex 等の外部 TCG カード DB API 連携**：今回のスコープ外
- **Discord OAuth2 によるクライアント側ログイン**：Jarvis は引き続き Google Identity Platform を一次認証とする（ADR-009 の NG1 と整合）
- **既存 Tickets Bot / John-Bot の置き換え**：ADR-009 のハイブリッド方針に従い共存させる
- **AI による Discord 自動返信機能**：別 ADR の領域

## 事業上の制約

### 制約 1: 情報秘匿の境界

クライアントに見せてよいのは「整形済み在庫一覧」と「クライアント自身の表示条件設定」のみ。以下はクライアントに**構造的に**不可視にする（権限制御だけでなく API レスポンスにも露出させない）：

- 仕入元の名称・存在・連絡先
- 解析プロンプト・解析モデル名・解析ログ
- 修正履歴・精度ログ
- 価格／数量チャート
- 商品マスタの編集権限

### 制約 2: GAS データ移行は Shingo の手作業が前提

パートナー Claude Code は Google アカウント配下のスプレッドシートに直接アクセスできない（セルフホストランナー上で動作し、Shingo の Google 認証を持たない）。したがって：

- 初期データ（`GPT_Knowledge` シート・仕入元別ルール等）は **Shingo が CSV エクスポートして Sales Anchor に取り込み可能な場所（リポジトリ or サーバー上の所定パス）に配置**する
- パートナー Claude Code は配置された CSV を読んで DB に投入するスクリプトを実装する
- 配置パス・CSV の列構成は実装時にパートナーが Shingo に提示し、Shingo が用意する

### 制約 3: 解析モデルのコスト・品質閾値は運用しながら決める

Gemini Flash Lite のコスト上限・許容できる解析精度の下限・上位モデルへの切り替え条件は、本 ADR 採択時点では未確定。本番運用で実測を取りながら PDCA で決める。パートナーは切り替え可能な抽象化を念頭に実装する。

### 制約 4: ADR-009（Discord Gateway）が前提

本 ADR は ADR-009 で規定された Discord Gateway 常駐 Bot 基盤の上で動作する。ADR-009 の実装が未完了の場合、パートナー Claude Code は ADR-014 実装の前提として ADR-009 の必要部分を併せて実装してよい。

### 制約 5: マルチテナント・既存セキュリティ基盤の遵守

新規テーブル・API は既存のマルチテナント分離（PostgreSQL RLS）、Pydantic 入力検証、監査ログ、JWT 認証の枠組みに統合する。これは既存 ADR 群（ADR-007 / ADR-009 / ADR-010）で確立済みの前提であり、パートナーが自律判断で守る範疇とする。

## パートナーに判断を委ねる事項

ADR-012 の役割分担に従い、以下の How はすべてパートナー Claude Code が自律判断する：

1. **既存 `products` / `product_inventory` テーブル（`docs/products_design.md`）と新規在庫管理モジュールの関係設計**：別系統並走か統合か部分連携か。最適と判断した形で設計する
2. **DB スキーマ・テーブル名・列構成・インデックス・RLS ポリシー**
3. **API ルーティング・Pydantic スキーマ・権限設計**
4. **改善ループの実装主体と起動方式**：Web Claude 手動分析 / Claude Code 定期バッチ / 両方の使い分け
5. **解析エンジンの抽象化（モデル切り替え可能性）とプロンプト構造**
6. **未知商品・低信頼度検知の閾値設計**
7. **チャートライブラリ・フロントエンド実装方式**
8. **Celery タスク粒度・再試行・冪等性設計**
9. **GAS データ移行スクリプトの設計と CSV 列構成の Shingo への提示**
10. **ADR-009 が未着手だった場合の Discord Gateway 実装範囲**

## 採用しない案

### 案 A: 既存 GAS 運用の継続改善

GAS のスクリプトを改善して精度を上げる。Sales Anchor とは API 連携で繋ぐ。

不採用理由：二重管理が解消されず、属人化・スクリプト破損リスクが残る。長期的には Sales Anchor 一本化が事業基盤として必要。

### 案 B: 高精度モデル（Claude Sonnet / GPT-4 系）一本での解析

最初から高精度モデルを採用してレビュー工数を最小化する。

不採用理由：仕入元メッセージは件数が多くコストが嵩む。コスト感度が高い領域なので、まず低コストモデルで運用しながら必要性を見極める方が事業判断として合理的。

### 案 C: 外部 TCG DB（TCGdex 等）の API を商品マスタとして使う

外部 DB を直接参照し、自前の商品マスタを持たない。

不採用理由：仕入元の表記揺れに対応しきれない、aliases や仕入元別ルールが持てない、外部 API の障害に運用が引きずられる。

## 結果

**期待される効果**

- GAS + スプレッドシート運用からの完全離脱
- 修正履歴が学習データとして再利用され、解析精度が運用しながら逓減する手動コスト
- クライアントから差別化の源泉（解析ロジック・仕入元情報）が構造的に隠蔽される
- 価格・数量の時系列可視化により相場観が定量化される
- CRM 側機能（見積・受発注）との連動余地が生まれる

**トレードオフ**

- 初期データ移行は Shingo の手作業が必要
- Gemini Flash Lite で品質が出ない場合のモデル切り替えコストが発生する可能性
- 既存 `products` との関係設計次第で軽微なリファクタリングが入る可能性

## Acceptance Criteria（事業価値ベース）

- AC-001: 仕入元 Discord チャンネルへの投稿が、自動で Sales Anchor の在庫マスタに反映される（高信頼度の場合）
- AC-002: 低信頼度の解析結果が Shingo の Discord に通知され、Sales Anchor 上で確認・修正できる
- AC-003: 修正履歴が蓄積され、後続の解析品質改善に再利用される仕組みが存在する
- AC-004: クライアントは、自分用に整形された在庫一覧のみを閲覧でき、解析ロジック・仕入元情報・チャート・精度ログにはアクセスできない
- AC-005: 管理者（Shingo）は、商品×状態別の価格・数量推移を時系列で確認できる
- AC-006: GAS の `GPT_Knowledge` 由来の商品マスタが、Shingo が提示した CSV から Sales Anchor に取り込まれている
