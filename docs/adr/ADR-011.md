## コンテキスト

salesanchorは以下の3者で開発される：

- **Shingo (PO)**: 事業判断と要件定義を担うが、コードレベルの技術判定はできない
- **Web Claude**: 設計・アーキテクチャ判断を担う相談相手。GitHubコネクタ経由でリポジトリ全体（ADR/Spec/実装コード）を参照可能
- **パートナーのClaude Code (Hikky-dev)**: セルフホストランナー上で実装・コードレビューを担う

このとき以下の課題が存在する：

1. 設計者(Web Claude)はコード実体を見ずに設計するため、現実乖離リスクがある
2. 実装者(Claude Code)は設計意図の解釈に迷っても、誰に質問すべきか経路がない
3. ShingoがブリッジになるがShingoは技術判定できないため、認識ズレが沈黙のまま蓄積する
4. ADR文書1種類だけでは「判断の根拠」と「実装指示」が混在し、Claude Codeの実装精度が落ちる
5. 従来のレビュー起動は手動のため、壁打ちサイクルが遅い
6. ADRの保存先が `docs/`・`docs/decisions/`・`docs/adr/` の3系統に散在している
7. パートナー側ランナーからfeedback.mdを返却する経路が未設計だった
   （ADR-001のレビューサイクルで実際に詰まった実績あり）

## 決定

### §1. ドキュメント2層構造

- **ADR (`docs/adr/ADR-XXX.md`)**: 判断の根拠を残す。「なぜそうしたか」のみ記述
- **Spec (`docs/spec/SPEC-XXX.md`)**: Claude Codeへの実装指示。受入基準は非技術者が判定可能な粒度で書く
  - 例: 「ログイン後3秒以内にダッシュボードが表示される」
  - 例: 「CSV出力時、文字化けなく全角文字が含まれるファイルが生成される」
- ADRとSpecは相互に参照IDを持つ（例: ADR-011 ↔ SPEC-011）
- Spec内の受入基準は AC-001, AC-002... と採番し、差分レポートで参照可能にする

### §2. レビューワークフローの自動トリガー化

**対象ファイル**: `.github/workflows/claude-pipeline.yml`

以下の変更をパートナー(Hikky-dev)に依頼する：

```yaml
on:
  push:
    branches: [main, develop]
    paths:
      - 'docs/adr/**.md'
      - 'docs/spec/**.md'
  workflow_dispatch:
    inputs:
      adr_files:
        description: 'レビュー対象ファイル（スペース区切り）'
        required: true
        type: string

concurrency:
  group: claude-pipeline-${{ github.ref }}
  cancel-in-progress: true
```

pushトリガー時は `inputs.adr_files` が存在しないため、変更ファイルをgit diffから自動検出する：

```yaml
- name: Detect changed ADR/Spec files (push event)
  id: detect
  if: github.event_name == 'push'
  run: |
    FILES=$(git diff --name-only HEAD~1 HEAD \
      | grep -E '^docs/(adr|spec)/.*\.md$' \
      | tr '\n' ' ')
    echo "adr_files=$FILES" >> $GITHUB_OUTPUT

- name: Set ADR_FILES env
  run: |
    if [ "${{ github.event_name }}" = "workflow_dispatch" ]; then
      echo "ADR_FILES=${{ inputs.adr_files }}" >> $GITHUB_ENV
    else
      echo "ADR_FILES=${{ steps.detect.outputs.adr_files }}" >> $GITHUB_ENV
    fi
```

### §3. フィードバック復路の確立

**Web Claude側（読み取り）**:
- ShingoはWeb Claude (claude.ai) のSettings > ConnectorsでGitHubコネクタを有効化し、
  本リポジトリへの読み取り権限を付与する
- これによりWeb Claudeはfeedback.md、既存ADR/Spec、実装コードを直接参照可能になり、
  設計時の現実乖離を抑制する

**パートナーClaude Code側（書き込み）**:
- レビュー完了後、`feedback/auto-YYYYMMDD-HHMM` ブランチを切ってfeedback.mdにコミット
- `gh pr create --base develop` でPRを自動作成（PR#280でHikky-devが実装済）
- ShingoがPRをdevelopにマージ → GitHubコネクタ経由でWeb Claudeが参照可能になる

**feedback.mdのフォーマット規約**:
本ファイル（feedback.md）の冒頭フォーマットをリファレンス実装として採用する：

```
# Feedback ログ
---
## YYYY-MM-DD — ADR-XXX レビュー (by Claude Code)
**対象**: `docs/adr/ADR-XXX.md`
**レビュー観点**: 内部整合性 / 既存資産との衝突 / 実装可能性 / 受入基準の明確性
### 総評
### Blocker（施行前に解消必要）
### Major（採否に影響する論点）
### Minor（品質）
### 良い点
### 推奨される次アクション
```

### §4. 質問返しチャネル

- Claude Codeが設計の解釈に迷った場合、**独自判断で実装を進めず**、
  `questions/QXX.md` に質問を書いて作業を停止する
- 停止時のgit状態: WIPコミットとして作業中ブランチに残す（stashは使わない）
- ShingoはそれをWeb Claudeに共有し、回答を対象ADR/SpecへのAmendmentコミットとして反映する

**再起動トリガー**:
対象ADR/Specが更新コミットされ、かつ該当questionファイルに
以下のフロントマターが付いたら、次回パストリガーで自動再評価する：

```yaml
---
status: answered
adr_id: ADR-XXX
answered_at: YYYY-MM-DD
---
```

### §5. 実装差分レポートの義務化

- 実装完了時、Claude Codeは Spec ↔ 実装の差分を `docs/diff-reports/DIFF-XXX.md` に記録する
- 受入基準ID単位で以下を明記する：
  - `AC-001: 完全実装`
  - `AC-002: 妥協 — 外部API制約により3秒→5秒に変更`
  - `AC-003: 未実装（理由: 次Sprint対応）`
- 差分があればShingoがWeb Claudeに共有し、必要に応じてADR追記またはSupersede

### §6. ADR/Spec起案の単一主担当

- ADR/Specの起案とpushは **ShingoのターミナルClaude Code経由に一元化**する
- Web Claude Project「ADR作成」はShingo不在時のフォールバック用途に格下げする
  （連番衝突回避のため平時は使用しない）
- 注: Web Claudeの外部ツール設定（ProjectのCustom Instructions等）の詳細管理は
  `docs/operational-runbook.md`（別途作成）で扱う

### §7. push経路とブランチ運用（ADR-010との整合）

ADR-010（main直push物理拒否）およびCLAUDE.mdの運用ルールに従い、
**ADR/Specも他のファイルと同じくPR経由でmainにマージする**。

運用フロー：
1. ShingoがTerminal Claude Codeで `feature/shingo/adr-NNN-topic` ブランチを作成
2. ADR-NNN.md をそのブランチにコミット
3. `gh pr create --base develop` でdevelop向けPRを作成
4. Shingoがマージ（Required approvals=0のため単独可）
5. develop → main は通常のリリースPRで

> **本§7はADR-001の§7「main直push許容」をSupersedeする。**
> ADR-010の部分Supersedeではなく、ADR/Specを他ファイルと同じ扱いに統一することで
> 例外ルールを設けずにADR-010と整合させる。

## 結果

**得られるもの**

- 設計→実装の認識ズレを検出する機構が成立する
- Web Claudeとパートナー Claude Code の壁打ちサイクルが **5〜10分単位** で回る
  （ランナーのキュー待ち・checkout・Claudeプロセス起動コストを含めた実測値ベース）
- 非技術者であるShingoは「コピペとAcceptを判断する係」に集中でき、技術的優劣の判定が不要になる
- Web Claudeがリポジトリ全体（ADR/Spec/実装コード）を参照できるようになり、
  設計が現実から乖離するリスクが下がる
- ADR/Specの保存先が一元化され、過去の意思決定が追跡可能になる

**トレードオフ**

- ADR/SpecもPR経由になるため、直pushより操作が1ステップ増える
  （ただしRequired approvals=0のため実質的な遅延は小さい）
- パートナーのClaude Code (Hikky-devのMac) の常時稼働が望ましい
  （非稼働時はワークフロー待機状態になり、5〜10分ではなく翌日になりうる）
- feedback.mdがdevelopブランチにあるため、ShingoがPRをマージするまで
  Web Claudeから参照できない（1〜2アクションの遅延）

**本ADR施行のために必要なアクション**

1. `docs/adr/ADR-001.md` を削除（本ADR-011に統合）
2. `docs/adr/README.md`（ADR Index）を新規作成
3. `docs/spec/`・`docs/diff-reports/`・`questions/` ディレクトリを作成（テンプレート配置）
4. パートナーに `claude-pipeline.yml` の §2 修正を依頼（concurrency + pushトリガー + input分岐）
5. `docs/BRANCH_PROTECTION_SETUP.md §4` に bypass記録を追記（commit `66675f8`、ADR-001初回push分）
6. `docs/ADR-009_discord_gateway.md` → `docs/adr/ADR-009-discord-gateway.md` 移動（次スプリント）
7. `docs/decisions/ADR-010-branch-protection.md` → `docs/adr/ADR-010-branch-protection.md` 移動（次スプリント）
