# ADR-082: フロントのみのデプロイで DB マイグレーション実行をスキップする

| 項目 | 内容 |
|------|------|
| ステータス | Accepted |
| 作成日 | 2026-05-30 |
| 起案 | Claude Code (Hikky-dev) |
| 関連 ADR | ADR-041（migration deploy 自動化）/ ADR-045（granted_scopes 自動実行・検証）/ ADR-036（DB ポカよけ）/ ADR-050（Release PR workflow）|

## What

本番デプロイ (`.github/workflows/deploy.yml`) の DB マイグレーション実行ステップ（part 1 / part 2）を、**当該デプロイにバックエンド／マイグレーション関連の変更が含まれない場合はスキップ**する。

具体的には、push to main の差分が以下のいずれにも該当しない場合（＝フロント / LP / ドキュメントのみの変更）、マイグレーション 2 ステップを `if:` 条件で skip する：

- `migrations/**`
- `scripts/**`
- `backend/**`
- `docker-compose.yml`
- `.github/workflows/deploy.yml`

ヘルスチェック・コンテナ状態確認・監視スタック確認・古いイメージの prune は**常時実行**へ分離し、マイグレーションが skip されても本番検証が抜けないようにする。

### 非対象

- マイグレーションの内容・順序・冪等性設計は一切変更しない（ステップを「実行するか否か」だけを条件化）。
- Docker イメージのビルド方式（`docker compose up -d --build`）は変更しない。フロント変更時のフロントイメージ再ビルドは本質的に必要なため。
- `pull_request` トリガーは追加しない（workflow-lint.yml の `no_pull_request` 規則を維持）。

## Why

直近のデプロイ実測（run 26680969807, CSS のみの変更）の所要時間内訳：

| ステップ | 所要 |
|---|---|
| Deploy (code + env + docker compose) | 111s |
| DB マイグレーション part 1 | 33s |
| DB マイグレーション part 2 + finalize | 69s |
| その他（LP build / rsync / verify） | 〜30s |
| **合計** | **約 4 分** |

マイグレーション part 1 / part 2 は約 60 個の **冪等（idempotent）ステップ**を**毎回フル実行**しており、フロントのみの変更（UI / CSS 修正など、QA 起点の頻出ケース）では「適用すべき新規スキーマ変更が何も無い」にもかかわらず約 100 秒を消費している。

マイグレーションは全て `IF NOT EXISTS` / `to_regclass` ガード / `ON CONFLICT DO NOTHING` 等で冪等に設計されているため、新規マイグレーションを含まないデプロイで実行しても「何も変わらない」。よってフロントのみのデプロイでは実行自体を省略しても本番状態は同一であり、安全に約 100 秒（全体の約 40%）を短縮できる。

## Scope / 安全性

- **保守的ゲート**: マイグレーションを skip するのは「変更が純粋にフロント / LP / ドキュメントのみ」のときだけ。`backend/**` や `docker-compose.yml`、`deploy.yml` 自身（＝新マイグレーションステップ追加）が変わった場合は必ず実行する。判定不能・差分取得失敗時は安全側（実行）に倒れる。
- **ポカよけ思想との整合**: 過去に「deploy.yml への migration ステップ追記漏れ → 本番 500」事故が複数回発生している（PR #518 等）。本 ADR は**追記漏れを増やさない**。新マイグレーション追加時は従来どおり deploy.yml にステップを足す運用は不変で、その変更自体がゲートにマッチしマイグレーションが実行される。
- **検証の常時実行**: ヘルスチェック（内部 + 外部 `Verify deployment`）・ADR-045 granted_scopes 検証はデプロイ種別に関わらず毎回走る。
- 判定には `dorny/paths-filter`（`test.yml` で実績あり・既存依存）を用いる。

## Consequences

- フロントのみのデプロイ: 約 4 分 → 約 2.5 分（マイグレーション 2 ステップ skip）。
- バックエンド / マイグレーションを含むデプロイ: 従来どおり全マイグレーション実行（所要は不変）。
- 将来の追加最適化候補（本 ADR 非対象）: CI でイメージをビルドして VPS は pull のみにする、変更サービスのみ再ビルドする等。効果は大きいが影響範囲も大きいため別 ADR で検討。
