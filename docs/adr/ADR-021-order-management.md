# ADR-021: 受注管理モジュール — フルフィルメント・売上計算・報酬計算の Sales Anchor 統合

> **Note**: 起案時点（2026-05-09）では ADR-019 として起案したが、main / develop では既に ADR-019（Meta審査期間中の英語UI一時デプロイ） / ADR-020（recording/english-ui 本番デプロイ実行）が確定済みのため、衝突回避のため ADR-021 に改番（2026-05-10）。

## ステータス
提案 (Proposed) — 2026-05-09 起案 / 2026-05-10 改番（ADR-019 → ADR-021）

## コンテキスト

### 現状の課題

Treasure Island JP / HIGH LIFE JPN は越境 EC（TCG 輸出）を主業として運用している。受注後のオペレーション（仕入れ・発送・売上計算・担当者報酬計算）は、独立した Google Apps Script 製アプリ「**OrderFlow Manager**」と Google Spreadsheet（158 列の単一シート）で運用されている。

リポジトリ外参考実装: `/Users/hitoshi/Documents/副業/業務委託/しんごさん/オーダー情報管理アプリ/`

OrderFlow Manager の機能カテゴリ:

| カテゴリ | 内容 |
|---|---|
| 取引状況 | ステータス、受注/営業担当、顧客、商品、請求書、決済 |
| 発送情報 | 住所、寸法、梱包、ラベル発行、配送業者、追跡番号、eLogi CSV 連携 |
| 仕入れ情報 | 仕入元、金額、数量、運送、ステータス |
| 予約販売情報 | デポジット、発売予定日 |
| トラブル情報 | 返品・返金処理 |
| 売上情報 | 売上高、原価、PayPal/WISE 手数料、報酬（営業/受注/発送/仕入/トラブル）、利益率 |
| 売上管理 | 月次集計、営業担当別 |

この運用には以下の限界がある:

1. **Sales Anchor（CRM）と分離**: 顧客・案件・受注情報は Sales Anchor、フルフィルメントは OrderFlow と二重管理。整合性を人間が担保する必要あり
2. **GAS / Spreadsheet スケール限界**: 158 列の単一シートは設計拡張・再利用・テストが困難
3. **報酬計算ロジックがシート上の数式に閉じている**: 営業/受注/発送/仕入/トラブル の 5 ロール × 売上 の報酬体系がスプレッドシート関数として実装されており、変更・監査・新規メンバ加入時の整備が手動
4. **モバイル / 現場運用での速度**: GAS HtmlService は速度・操作性に限界、特に発送現場でのスマホ操作で遅延
5. **データ分析・ダッシュボード化困難**: Spreadsheet 集計を超える可視化（時系列、担当者別、商品別）が困難

### なぜ Sales Anchor 内で実装するのか（事業価値）

- **CRM と業務オペレーションの一元化**: 受注 → 仕入 → 発送 → 売上 → 報酬 が 1 つのデータモデル上で完結
- **報酬計算の透明化**: 計算ロジックを ADR + コード + テストで管理、過去報酬の再計算・監査も可能
- **モバイル UX**: React + Vite ベースの SPA で発送現場の操作速度を改善
- **データ基盤の活用**: PostgreSQL に蓄積したデータをダッシュボード・レポート・他モジュールから再利用可能
- **OrderFlow からの完全離脱（移行完了後）**: 属人化・スプレッドシート破損リスク・拡張限界の解消

## 決定 — What を実現するか

Sales Anchor 内に **受注管理（OrderManagement / Fulfillment）モジュール** を実装する。実現したい価値は以下のとおり。

### 1. 受注一覧と統合管理

OrderFlow の「取引状況」相当のオーダー一覧を Sales Anchor 上で表示・操作する。既存 `orders` テーブルを起点とし、フルフィルメント関連情報（仕入・発送・売上・トラブル）を関連エンティティとして紐付けて表示する。

- ステータスフィルタ（未処理 / 仕入中 / 配送中 / 完了 / トラブル / キャンセル）
- キーワード検索（顧客名・商品名・請求書番号・追跡番号・リード ID）
- ソート（受注日 / 金額 / ステータス / 担当者）
- グループ別件数表示（OrderFlow `calculateGroupCounts_` 相当）

### 2. 仕入れ情報の登録と進捗管理

受注に紐づく仕入れ情報を Sales Anchor から登録できる。

- 仕入担当者、仕入日、仕入元、取引番号、金額、数量、送料、運送会社、追跡番号、ステータス
- 仕入元検索（ADR-014 で構築する在庫マスタと連携可、または独立の suppliers マスタ参照）
- 仕入確定 → ステータス遷移（仕入れ → 配送中）
- 仕入元別の取引履歴閲覧

### 3. 発送情報の登録と外部システム連携

受注に紐づく発送情報を登録し、外部の発送代行システムと連携する。

- 受取人情報（住所・電話・メール・国・ZIP）
- 寸法・重量・梱包情報・HS コード・税関情報
- 発送ラベル発行（ステータス遷移）
- **eLogi CSV 出力**: 既存 OrderFlow と同じフォーマットで CSV 生成、Drive 等の指定先に保存
- ラベル PDF アップロード（運送会社発行ラベルを保管）
- 追跡番号の登録、配送業者選択

### 4. 売上計算とレポート

受注ごとに売上情報を計算・記録する。OrderFlow の売上情報カテゴリ（27 列）を構造化テーブルに分解する。

- 売上高（仕入原価との関係で原価率・利益率を自動計算）
- 決済手数料（PayPal / WISE / 為替手数料）
- 外注費・荷造運賃・広告費・返送料・返金額
- インセンティブ計算用フィールド
- 営業利益・営業利益率
- 消費税還付込・消費税還付
- 月次集計（売上管理シート相当：担当者別 × 月別）

### 5. 担当者報酬計算

5 ロール × 売上 の報酬計算を Sales Anchor 上で実施する。

- 5 ロール: 営業 / 受注 / 発送 / 仕入 / トラブル
- 各ロールの報酬式（売上額・固定額・歩合 等）はテナント設定として保存
- **テナント（クライアント）ごとに掛け率が異なるため、管理者ユーザーが画面から計算式を編集できる UI を提供する**（Shingo 確定 2026-05-10）
- 受注ごとに各ロール担当者を割当（既存 `staff` 参照）
- 月次・担当者別の報酬レポート出力

### 6. トラブル情報の記録と返金処理

返品・返金トラブルを構造化して記録する。

- トラブル発生日、理由、対応方法、返送方法、返金額・返金方法、返送ラベル
- 在庫戻し（ADR-014 在庫管理との連携）
- 売上計算への反映（返金 → 売上から差し引き）

### 7. 予約販売情報

通常受注と分離した予約販売（デポジット型）の管理。

- 請求書番号、発行日、支払期日、支払確認日
- 発売予定日、デポジット率、確定数
- 通常受注への遷移（発売後に正規受注として処理）

### 8. Spreadsheet からの段階的データ移行

既存 OrderFlow Spreadsheet データを Sales Anchor に取り込む。

- **過去データ全件の取り込み**（Shingo 確定 2026-05-10、過去受注・売上・報酬の完全保全）
- Sales Anchor service account / 移行スクリプト実行ユーザーへ Spreadsheet 読み取り権限を直接付与（CSV エクスポート方式は採らない、Drive API / Sheets API で直読）
- 並行運用期（Spreadsheet と Sales Anchor が両方稼働）の整合性確保戦略
- 完全移行のタイミング判断と OrderFlow 廃止条件

## Scope 外

以下は本 ADR では扱わない:

- **CRM 側の受注ライフサイクル変更**: 既存 `orders` の作成・更新フローは大きく変えない。OrderManagement モジュールは既存 orders を拡張する形で構築
- **会計システム連携**: freee / マネーフォワード等への自動連携は別 ADR
- **配送料率の自動取得**: 各キャリアの料金表からの自動見積は別 ADR
- **Sales Anchor 既存ダッシュボードの全面再設計**: 受注管理用ダッシュボードのみ追加、既存 dashboard はそのまま
- **元 OrderFlow の GAS コードのそのまま移植**: 参考実装として `docs/reference/orderflow-manager/` に保管されるが、Sales Anchor の React + FastAPI に **再実装** する
- **OrderFlow Manager の機能改善**: 移行完了までは OrderFlow は OrderFlow として動かす、改修しない

## 事業上の制約

### 制約 1: 既存 `orders` テーブルの後方互換性

`orders` は既に Phase 1-B-2 Step 5d で `company_id` + `contact_id` 必須化された B2B モデル唯一の正。本 ADR の追加機能は **既存 `orders` の構造を破壊しない**。拡張カラムまたは関連テーブルとして実装する。

### 制約 2: マルチテナント・既存セキュリティ基盤の遵守

新規テーブル・API は既存のマルチテナント分離（PostgreSQL RLS）、Pydantic 入力検証、監査ログ、JWT 認証の枠組みに統合する。これは既存 ADR 群（ADR-007 / ADR-009 / ADR-010）で確立済みの前提であり、パートナーが自律判断で守る範疇とする。

### 制約 3: 報酬計算の透明性と監査可能性

報酬計算は **テストコードでロジックを保証**し、月次レポートで個人ごとに「いつ・どの受注から・いくら発生したか」を再現可能にする。担当者から監査要請があった時に元データを示せる構造にする。

### 制約 4: Spreadsheet 過去データを全件 Sales Anchor に移行する

既存 OrderFlow Spreadsheet の **過去データ全件を Sales Anchor に移行する**ことを最終目標とする（Shingo 確定 2026-05-10、新規のみ移行ではなく履歴も含めた完全移行）。実装は一括ではなく段階的に進めるが、最終的には Sales Anchor が単一の正となり OrderFlow Spreadsheet は廃止する。並行運用期間中は重複入力ガードを設ける。

Spreadsheet 読み取りアクセスは、CSV エクスポートではなく **Sales Anchor の service account / 移行スクリプト実行ユーザーへ直接権限付与**（Drive API / Sheets API で直読）とする（Shingo 確定 2026-05-10）。

### 制約 5: eLogi CSV 互換性 + 発送代行プラガビリティ

現行のメイン発送代行は **eLogi**。OrderFlow の eLogi CSV 出力（19 カラムの固定フォーマット）は eLogi 側の受け取り仕様と整合しており、Sales Anchor からの出力も **同フォーマットを維持** する。

加えて、**将来的に発送代行サービスが追加される含み**を持たせる（Shingo 確定 2026-05-10）。本 ADR では eLogi を第一実装とするが、出力アダプタ層を抽象化し、追加キャリア（DHL / FedEx / ヤマト等）の CSV / API 連携を別 ADR で個別に追加できる設計とする。eLogi 以外の追加実装は本 ADR の Scope 外。

### 制約 6: Meta App Review との独立性

本モジュールは Meta App Review の直接審査対象外（Facebook/Instagram 連携機能と独立）。並行で進めて問題ないが、reviewer がテストアカウントでアプリを探索する時に **新規追加 UI が破綻していない**ことは保証する。

## パートナーに判断を委ねる事項

ADR-012 の役割分担に従い、以下の How はすべてパートナー Claude Code が自律判断する:

1. **既存 `orders` テーブルとの関係設計**: 拡張カラム追加 / 関連テーブル新設 / 別系統並走 のどれが最適か
2. **DB スキーマ・テーブル名・列構成・インデックス・RLS ポリシー**: OrderFlow の 158 列を正規化した結果のテーブル数と分割粒度
3. **API ルーティング・Pydantic スキーマ・権限設計**: REST endpoint 設計、staff role ベースの認可
4. **ステータス遷移の State Machine 設計**: 既存 `OrderStatus` enum との互換維持または拡張
5. **報酬計算式の構造化方法**: 計算ロジックを Python コード固定 / DB に式を保存 / DSL / プラガブル抽象化のどれが適切か。**Shingo 確定 2026-05-10: クライアント（テナント）ごとに 5 ロールの掛け率が異なるため、ユーザー（管理者）が画面から計算式を編集できる構造が必須**（コード固定は不可）
6. **eLogi CSV 出力の実装方式**: バックエンドで生成 / フロントで生成 / Cloud Function 切り出し のどれが運用上良いか
7. **ラベル PDF / CSV の保存先**: VPS local / Google Drive / S3 / 外部ストレージのどれを採用するか
8. **Spreadsheet データ移行スクリプトの設計と実行戦略**: 一括 import / 増分 sync / 並行運用支援ツール のどれを優先実装するか
9. **トラブル処理が在庫管理（ADR-014）に反映される導線設計**: 自動連携 or 手動マーキング
10. **予約販売を通常受注テーブルに統合するか別テーブルにするか**
11. **チャートライブラリ・フロントエンド実装方式**: 既存 Sales Anchor で使われている技術スタックに合わせる
12. **モバイル UX の最適化方針**: PWA / レスポンシブ / 専用モバイル UI のどれを優先するか
13. **既存 `purchase_order` / `shipping` / `supplier` の活用範囲**: 本 ADR で扱う仕入・発送情報をこれらのテーブルに含めるか、専用テーブルを新設するか
14. **段階リリース戦略**: ~~受注一覧 → 仕入 → 発送 → 売上 → 報酬 のどの順序で MVP を出すか~~ **Shingo 確定 2026-05-10: `受注一覧 → 売上 → 発送 → 仕入 → 報酬` の順で MVP を出す**。発送実装が完了したタイミングでモバイル UX 対応（PWA / レスポンシブ等の選択はパートナー判断）に着手する。トラブル処理の在庫戻し連携は仕入実装後の Phase 2（推奨案）として ADR-014 と接続する

## 採用しない案

### 案 A: OrderFlow Manager（GAS）の継続運用

GAS 側を改善し、Sales Anchor とは API 連携で繋ぐ。

不採用理由: 二重管理が解消されない、GAS の実装拡張限界、報酬計算ロジックの透明化が進まない、長期的には Sales Anchor 一本化が事業基盤として必要。

### 案 B: OrderFlow の完全移植を一気に行う

既存 OrderFlow の全機能（158 列・全 UI・全 GAS ロジック）を一括で Sales Anchor に再実装し、Spreadsheet を即廃止する。

不採用理由: 移行リスクが高すぎる。本番運用中の業務システムを一気に切り替えるのは、データロス・運用停止・既存ユーザの慣れ消失のリスクを伴う。段階リリースの方が事業継続性が高い。

### 案 C: 受注管理を別アプリとして独立構築

Sales Anchor とは別の Web アプリとして受注管理を作り、API でデータ連携する。

不採用理由: マイクロサービス化のメリットより、CRM と業務オペレーションを 1 アプリに統合する方が運用コストと UX が良い。既存 `orders` / `staff` / `customers` 等の資産を直接参照できる。

### 案 D: 報酬計算ロジックをコード固定（DB に保存しない）

5 ロールの報酬計算式を Python コードに書き、テナントごとの設定変更は不可とする。

不採用理由: 担当者ロール構成や歩合率はテナント運用上変わる可能性が高い（新メンバ加入、ロール体系変更等）。コード固定だと毎回コード変更 + デプロイが発生し運用負担が高い。**Shingo 確定 2026-05-10: クライアント（テナント）ごとに掛け率が異なるため、ユーザー（管理者）自身が画面から編集できることが要件**。コード固定はこの要件を満たせない。

### 案 E: 売上計算を Sales Anchor 外（freee 等会計システム）に委ねる

売上情報は会計システムの責任範囲とし、Sales Anchor は受注・発送までを管理する。

不採用理由: 報酬計算（5 ロール × 売上）が成立しない。報酬計算には粗利・諸経費を含む詳細な売上データが必要だが、会計システムから個別取引単位で取得するのは API 制約上困難。Sales Anchor 内に売上データを持つ方が報酬計算と直結する。

## 結果

**期待される効果**

- 受注 → 仕入 → 発送 → 売上 → 報酬 の業務フローが Sales Anchor 一元化
- OrderFlow Manager の段階的廃止 → 二重管理コスト削減・属人化解消
- 報酬計算ロジックの透明化と監査可能性の確保
- モバイル UX 改善（特に発送現場）
- データ基盤統合によるダッシュボード・レポート機能拡充の余地

**トレードオフ**

- Spreadsheet からの移行は段階的に進める必要があり、運用負荷が一時期増える
- 既存 `orders` / `shipping` / `purchase_order` の関係設計が複雑化、初期実装で慎重な判断が必要
- 報酬計算ロジックを DB 駆動にすると初期設計コストが上がる（が、長期保守性で回収）

## Acceptance Criteria（事業価値ベース）

- AC-001: Sales Anchor 上で OrderFlow の取引状況相当の受注一覧が表示でき、ステータスフィルタ・キーワード検索・ソートが機能する
- AC-002: 受注に紐づく仕入れ情報（仕入元・日・金額・数量・追跡番号・ステータス）を Sales Anchor から登録・編集でき、ステータス遷移と連動する
- AC-003: 受注に紐づく発送情報を Sales Anchor から登録でき、eLogi CSV 出力が既存フォーマットで生成される
- AC-004: 売上情報（売上高・諸経費・利益率）が受注ごとに計算され、月次レポートとして担当者別・商品別に集計できる
- AC-005: 担当者報酬計算（5 ロール × 売上）が Sales Anchor 上で実行され、月次・担当者別レポートに反映される。**テナントごとの掛け率を管理者ユーザーが画面から編集できる**
- AC-006: トラブル情報（返品・返金）が記録でき、売上計算と在庫管理（ADR-014）に反映される（自動連携は Phase 2、Phase 1 は手動マーキング）
- AC-007: 予約販売（デポジット型）の管理ができ、発売後に通常受注へ遷移する
- AC-008: 既存 OrderFlow Spreadsheet からの **過去データ全件**取り込みスクリプトが提供される（Drive API / Sheets API 経由で直読）
- AC-009: 段階リリース順 `受注一覧 → 売上 → 発送 → 仕入 → 報酬` を満たす構造。発送実装完了タイミングでモバイル UX 対応（PWA / レスポンシブ等）に着手できる
