# Zero Trust ポリシー - Sales Anchor

## 1. Zero Trust の原則

> 「何も信頼せず、常に検証する」

全てのアクセスは、ネットワーク上の位置に関係なく、認証・認可・暗号化を経て初めて許可される。

## 2. 現在の実装状況

### 認証レイヤー（実装済み）

| 原則 | 実装 | 状態 |
|------|------|------|
| 全リクエスト認証 | Firebase JWT検証（全APIエンドポイント） | ✅ |
| 多要素認証 | Firebase MFA対応 | ✅ |
| トークン失効 | Redisブラックリスト（ログアウト時即時無効化） | ✅ |
| 最小権限 | admin/userロール分離 | ✅ |
| セッション制限 | JWT TTL 1時間 + キャッシュ5分 | ✅ |

### ネットワーク分離（実装済み）

| 原則 | 実装 | 状態 |
|------|------|------|
| マイクロセグメンテーション | Docker frontnet/backnet分離 | ✅ |
| DB外部非公開 | PostgreSQL/Redisはbacknetのみ | ✅ |
| 暗号化通信 | TLS 1.2+ 強制、HSTS有効 | ✅ |
| DDoS防御 | Cloudflareプロキシ | ✅ |
| ファイアウォール | UFW（22/80/443のみ） | ✅ |

### データ保護（実装済み）

| 原則 | 実装 | 状態 |
|------|------|------|
| テナント分離 | PostgreSQLスキーマ分離 + RLS | ✅ |
| IDOR防止 | tenant_idはDB参照（JWT/URL非依存） | ✅ |
| 入力検証 | Pydanticバリデーション（全入力） | ✅ |
| 監査ログ | 全CRUD操作・認証イベント記録 | ✅ |

### コンテナセキュリティ（実装済み）

| 原則 | 実装 | 状態 |
|------|------|------|
| 非rootユーザー | 全コンテナでappuser実行 | ✅ |
| 権限昇格防止 | no-new-privileges:true | ✅ |
| リソース制限 | メモリ・CPU上限設定 | ✅ |
| 読み取り専用FS | frontend/Redisでread_only | ✅ |
| ヘルスチェック | 全サービスにhealthcheck定義 | ✅ |

## 3. 今後の強化計画

### 短期（3ヶ月以内）

| 施策 | 概要 | 優先度 |
|------|------|--------|
| サービス間mTLS | コンテナ間通信の相互TLS認証 | High |
| ネットワークポリシー | Kubernetes移行時にNetworkPolicy適用 | Medium |
| シークレット管理 | HashiCorp Vault導入 | Medium |

### 中期（6ヶ月以内）

| 施策 | 概要 | 優先度 |
|------|------|--------|
| SIEM統合 | セキュリティイベントの一元管理 | Medium |
| 動的認可 | OPA/Regoによるポリシーベース認可 | Low |
| コンテナイメージ署名 | Cosignによるイメージ検証 | Low |

### 長期（1年以内）

| 施策 | 概要 | 優先度 |
|------|------|--------|
| サービスメッシュ | Istio/Linkerd導入 | Low |
| ゼロトラストネットワークアクセス | BeyondCorp相当の実装 | Low |
| CSPM導入 | クラウドセキュリティ態勢管理 | Low |
