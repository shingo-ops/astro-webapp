# インシデント対応手順書 - Jarvis CRM

## 1. インシデント分類

| レベル | 定義 | 対応時間 | エスカレーション |
|--------|------|---------|----------------|
| P1 (Critical) | サービス全停止、データ漏洩 | 即時 | 全チーム招集 |
| P2 (High) | 主要機能障害、セキュリティ侵害の疑い | 1時間以内 | 開発リーダー通知 |
| P3 (Medium) | 一部機能障害、パフォーマンス低下 | 4時間以内 | 担当者対応 |
| P4 (Low) | 軽微なバグ、UIの不具合 | 翌営業日 | チケット起票 |

## 2. インシデント対応フロー

### ステップ1: 検知と初動対応

```
1. アラート受信（Grafana/Uptime Kuma/ユーザー報告）
2. インシデントレベルを判定
3. 対応チームを招集
4. インシデントチャンネルを開設（Slack等）
5. タイムラインの記録を開始
```

### ステップ2: 封じ込め

```
【サービス障害の場合】
- docker compose ps でコンテナ状態確認
- docker compose logs <service> でエラーログ確認
- 必要に応じてサービス再起動

【セキュリティ侵害の場合】
- 影響範囲の特定（どのテナント、どのデータ）
- 不正アクセス元IPのブロック（UFW）
- 侵害されたアカウントの無効化
- 証拠の保全（ログのバックアップ）
```

### ステップ3: 根本原因分析

```
1. ログ分析（Loki/PostgreSQL audit_logs）
2. 変更履歴の確認（git log）
3. 外部要因の調査（Cloudflare WAFログ）
4. タイムラインの作成
5. 根本原因の特定
```

### ステップ4: 復旧

```
1. 修正パッチの作成とテスト
2. ステージング環境での検証（可能な場合）
3. 本番デプロイ
4. 復旧確認（ヘルスチェック、機能テスト）
5. 影響を受けたテナントへの通知
```

### ステップ5: 事後対応

```
1. インシデントレポートの作成
2. 再発防止策の策定
3. 対策の実装スケジュール決定
4. 関係者への報告
5. ドキュメントの更新
```

## 3. GDPR対応（個人データ漏洩時）

| 項目 | 期限 | 対応 |
|------|------|------|
| 監督当局への通知 | 72時間以内 | 漏洩の範囲・影響・対策を報告 |
| データ主体への通知 | リスクが高い場合は速やかに | 平易な言語で通知 |
| 記録の保管 | 恒久的 | インシデントの詳細を記録 |

## 4. 緊急コマンド集

### サービス状態確認
```bash
docker compose ps
docker compose logs --tail=100 backend
docker compose logs --tail=100 nginx
```

### 緊急再起動
```bash
docker compose restart backend
docker compose restart nginx
```

### 不正アクセス元ブロック
```bash
sudo ufw deny from <IP_ADDRESS>
```

### データベースバックアップ（緊急）
```bash
docker compose exec postgres pg_dump -U jarvis jarvis_db | gzip > emergency_backup_$(date +%Y%m%d_%H%M%S).sql.gz
```

### 全サービス停止（最終手段）
```bash
docker compose down
```
