1サマリー

このADRで決めたいこと ① claude-pipeline の順序を Generator → Evaluator → Reviewer から Generator → Reviewer → Evaluator に逆転
② PR フロー正常化: PR 作成は常に Generator (役割)、人間は PR 作成しない (merge のみ)、Reviewer/Evaluator は gh pr review でレビュー専念
③ status name を新体系へ rename (evaluator_passed 等の legacy 名を整理)
④ 修正サイクル: Reviewer/Evaluator が FAIL → Generator が同じ PR ブランチに修正 push + 再レビュー依頼 → Reviewer 再走 → (画面修正あれば) Evaluator 再走 → 両方 PASS で merge ready
⑤ Evaluator 2 層化: Layer 1 (Playwright、default) / Layer 2 (Claude in Chrome、Meta OAuth・本番セッション・撮影前等) / Skip (画面修正不要時)
⑥ 最終 merge は人間 (ひとし or しんご) が gh pr merge --squash
⑦ 移行期間 (1 ヶ月) は旧 status name を alias で読み込み互換

2背景

2-1. 役割明確化と順序の不整合

ADR-042 のドラフト議論で Reviewer / Evaluator の役割を以下に確定しました:

段階エージェント役割静的/動的
Loop 1 (先)ReviewerGenerator が作成したコードのレビュー (静的読解)静的、ブラウザ不要
Loop 2 (後)EvaluatorReviewer 通過後の UI/UX をブラウザで評価 (Playwright)動的、ブラウザ必須

しかし、現在の claude-pipeline state machine は 逆順で実装されています:

  • ~/.claude/agents/evaluator.md line 8: "4-stage pipeline (Planner → Generator → Evaluator → Reviewer)"
  • ~/.claude/agents/reviewer.md line 10: "last stage of the 4-stage pipeline" + status evaluator_passed を起動条件
  • .github/workflows/claude-pipeline.yml の job 順序も同様

2-2. なぜ順序逆転が必要か (現状の問題点)

項目現状 (Evaluator 先)新方式 (Reviewer 先)
役割定義との整合不整合 役割は「Reviewer = コード、Evaluator = UI/UX」と定義したが順序が逆整合
早期 failコードが汚くても先に重い Playwright を回す (無駄)コードレビュー段階で fail → Playwright 回避 (時間・コスト節約)
Reviewer 修正後の Evaluator 再実行順序的に発生しにくいが、Reviewer が「動かないコード」を見ても fix できないReviewer 通過 = コード合格 → Evaluator が動的検証、両方 PASS で完成
ループ構造の明快さ不明瞭 (役割と順序がズレているため)明快 (静的 → 動的の段階的検証)

3-3. PR フローと役割分担

操作旧 (claude-pipeline 独自)新 (GitHub 標準)
PR 作成 (gh pr create)Reviewer (最終 gate のため)Generator (実装完了時、GitHub 標準と同じ)

Reviewer の操作PR 作成 + push + status 操作gh pr review --approve or --request-changes のみ (PR は触らない)

修正サイクルReviewer が CHANGES_REQUESTED → Generator が新規 commit → Reviewer 再走Reviewer or Evaluator が --request-changes → Generator が同じ PR ブランチに push → Reviewer 再走 → Evaluator 再走
最終 mergeReviewer が PR 作成、人間が merge ボタン両方 approve 後、人間 (ひとし or しんご) が gh pr merge --squash

3-4. なぜ「PR 作成 = Generator」が正しいか

  1. GitHub 標準フローと一致 — オープンソース / 業界標準では「開発者が PR 作成、レビュアーが review、メンテナが merge」
  2. 役割の単一責任 — Reviewer = コードレビューだけ、Evaluator = UI/UX だけ。PR の作成/merge は別軸の操作
  3. Mode B (External PR review) と統一 — 既存の Mode B も Reviewer は gh pr review のみ。Mode A も同じパターンに揃える
  4. 修正サイクルの自然さ — Generator は自分の PR を持つ → 修正を push → CI 再走 → レビュアー再 approve、という GitHub ネイティブの流れ
  5. 権限分離 — レビュアーが push 権を持たないことで「レビュアーが勝手に書き換える」事故を防ぐ

5What (実装内容詳細)

5-1. Generator エージェント変更 (PR 作成責務追加)

workflow 変更 (PR 作成 step 追加) 
# 現状
Step Final: commit + write status `in_progress` → ハンドオフ (Reviewer は別ステップで push + PR 作成)

# 新
Step Final:
  1. commit changes
  2. git push origin HEAD  (feature ブランチ作業 push)
  3. gh pr create --base develop ...  🆕 (Generator が PR を open)
  4. PR URL を sprint メタに記録
  5. write status `in_progress` (Reviewer 起動条件)

# 修正サイクル時
- Reviewer / Evaluator が CHANGES_REQUESTED → Generator が同じ feature ブランチに修正 push
- PR は自動更新される (新規 PR は作らない)
- status を `in_progress` に戻す → Reviewer 再走

5-2. Reviewer エージェント変更

frontmatter description (現状 → 新) 
# 現状
description: ... after the Evaluator has PASSED a sprint, audit the Generator's code and APPROVE (push + open PR) ...

# 新
description: ... the FIRST review stage immediately after the Generator opens a PR. ...
  ... You audit the Generator's CODE (static reading, no browser).
  ... APPROVE via gh pr review --approve (Generator's PR advances to Evaluator).
  ... CHANGES_REQUESTED via gh pr review --request-changes (Generator revises in-place).
  ... You do NOT create/push/merge PRs (those are Generator's and human's responsibility).
Mode A workflow 変更 (大幅簡素化) 
# 現状 (Reviewer が push + PR 作成、責務肥大)
Step 1: Read sprint status — must be `evaluator_passed`
Step 6: If APPROVED → push + gh pr create + write `approved`
Step 6: If CHANGES_REQUESTED → write `changes_requested`

# 新 (Reviewer は PR review のみ、Mode B と同じパターン)
Step 1: Read sprint status — must be `in_progress` AND PR URL exists in sprint meta
Step 2: gh pr view → 差分取得 → コードレビュー
Step 3-5: 既存と同じ (10 項目 rubric で audit)
Step 6: If APPROVED → gh pr review --approve + write status `reviewer_approved`
Step 6: If CHANGES_REQUESTED → gh pr review --request-changes + write `changes_requested` + fail_stage=reviewer
  # PR は触らない、merge も触らない

5-3. Evaluator エージェント変更

frontmatter description (現状 → 新) 
# 現状
description: ... after the Generator has completed a sprint and written its report ...

# 新
description: ... after the Reviewer has APPROVED the PR (code review passed). ...
  ... The Evaluator is the SECOND review stage — UI/UX browser evaluation via Playwright.
  ... APPROVE via gh pr review --approve (PR ready for human to merge).
  ... CHANGES_REQUESTED via gh pr review --request-changes (Generator revises, Reviewer re-runs).
  ... You do NOT create/push/merge PRs.
workflow 変更 (PR review に簡素化) 
# 現状
Step 1: Read sprint status — must be `in_progress`
Step Done: Write `evaluator_passed`, hand off to Reviewer (Reviewer がその後 push/PR)

# 新
Step 1: Read sprint status — must be `reviewer_approved` AND PR URL exists
Step 2: Playwright で AC 検証 (既存と同じ)
Step Done (PASS):
  gh pr review --approve on PR
  Write status `evaluator_passed` (両方 PASS、merge ready)
  Discord 通知: "PR #N ready for human merge"
Step Done (FAIL):
  gh pr review --request-changes with bug report
  Write status `changes_requested` + fail_stage=evaluator
  # PR は触らない、merge も触らない (人間が merge)

5-3. claude-pipeline.yml 変更

# 旧 job 順序
jobs:
  generator:
    ...
  evaluator:
    needs: generator
    ...
  reviewer:
    needs: evaluator

6Why

  1. 役割定義 (ADR-042 + agents-role-clarification) との整合 — Reviewer = コード (静的)、Evaluator = UI/UX (動的) の順序は静的 → 動的が自然
  2. 早期 fail で重い処理を回避 — コードレビュー段階で fail なら Playwright (動的検証) を回さない、CI コスト・時間削減
  3. 修正サイクルの効率化 — コード品質 fail → Generator 修正 → 再 Reviewer (軽い)、最終的に UI/UX 検証へ
  4. ループ構造の明確化 — Loop 1 (静的) → Loop 2 (動的) の段階的検証が直感的
  5. GitHub 標準フロー準拠 — 「開発者 (Generator) が PR 作成、レビュアー (Reviewer/Evaluator) がレビュー、メンテナ (人間) が merge」という OSS / 業界標準パターンに揃える
  6. 役割の単一責任化 — Reviewer = コードレビューだけ、Evaluator = UI/UX だけ、PR 作成 = Generator だけ、merge = 人間だけ。各役割が単一責任で設計が明快
  7. Mode B (External PR review) と統一 — Reviewer の既存 Mode B (外部 PR レビュー) は gh pr review のみで動作。Mode A も同パターンに揃えれば実装も統一
  8. Evaluator スキップで重い処理を更に削減 — 画面修正不要な変更 (backend のみ、docs、リファクタ) では Playwright を回さない、Reviewer のみで完結

6-1. Evaluator method 選択基準 (2 層化対応)

変更の種類Layer 1 (Playwright)Layer 2 (Claude in Chrome)判定理由
frontend/src/ の component / page / hook / style 変更必須不要 (通常)標準 AC 検証で十分
frontend の i18n テキスト変更 (ja.json / en.json)必須—表示変わるため
backend API 仕様変更 (request/response 形式)必須—frontend 連携影響、UI 動作変わる可能性
Meta OAuth / Webhook 関連 (ADR-018, ADR-026, ADR-041 系)推奨必須実 Meta セッションでないと検証困難
本番デプロイ後の目視確認—必須ログイン済 Chrome 必須
Meta App Review 撮影前確認—必須GIF 録画 + 実ブラウザ必須
外部システム連携 (GCP/Cloudflare/Firebase live セッション)—必須実セッション必須
backend 内部実装のみ (API 仕様不変)スキップ可—UI 不変
migration / schema 変更 (UI 影響なし)条件付きスキップ可—影響あれば Layer 1 必須
テストコード追加・修正のみスキップスキップUI に影響なし
docs / runbook 変更スキップスキップUI に影響なし
CI workflow / config 変更スキップスキップUI に影響なし
リファクタ (機能変化なし)スキップ可—振る舞い不変
method 宣言の責任 Generator (or 人間 Generator 役) が PR 本文の ## Evaluator method セクションで Layer 1 / Layer 2 / Skip を明示宣言する。 Reviewer は宣言が妥当か判定:
  • 不当な Skip (frontend 変更あるのに Skip 宣言) → CHANGES_REQUESTED で撤回
  • Layer 2 のみ宣言だが標準 AC test 含まれる → Layer 1 も必須化を要求
  • Meta OAuth 系で Layer 2 未宣言 → Layer 2 追加を要求
最終的に Reviewer approve = Evaluator method の合意。
