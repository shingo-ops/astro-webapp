3新運用の全体像

ADR 起案は基本的にしんごさんが行う (パターン A・B 共通)。パターン A はしんごさん本人がすべて完結、パターン B は claude-pipeline 経由で Plan をひとし側に届けてひとしの Generator が実装する。Reviewer と Evaluator は順序のあるループ: ① Generator ⇄ Reviewer (コードレビュー) ループで PASS → ② Generator ⇄ Evaluator (UI/UX ブラウザ評価) ループで PASS → approve。Reviewer/Evaluator はそれぞれ系統別 (しんご系 / ひとし系) に持つ。

役割の明確化 Reviewer = Generator が作成したコードのレビュー (コードレベル)
Evaluator = Reviewer 通過後に UI/UX をブラウザを使って評価 (Playwright 等)
ループの構造 (2 系統、順序あり) ① Generator ⇄ Reviewer ループ — コードの品質チェック、FAIL なら Generator が修正 push、PASS で次へ
② Generator ⇄ Evaluator ループ — UI/UX 動作確認、FAIL なら Generator が修正 push、PASS で approve
※ Loop 1 と Loop 2 は順次実行 (Reviewer 完了後に Evaluator)
※ Evaluator スキップ条件: 画面 (UI/UX) 修正が不要な変更 (backend のみ、ドキュメント、リファクタ等) は Evaluator スキップ可
PR フロー (GitHub 標準準拠) ① PR 作成 = 常に Generator (役割名。人間が Generator 役の場合も同じ。人間が直接 PR は作らない)
② 再レビュー依頼 = Generator が修正 push 後に再要求 (Reviewer/Evaluator は反応するだけ)
③ Reviewer/Evaluator の操作は gh pr review --approve or --request-changes のみ
④ 最終 merge = 人間 (ひとし or しんご) が gh pr merge --squash
パターンの違い ① パターン A: しんごさん本人が完結 (起案 + 本人実装 + しんご Reviewer + しんご Evaluator)
② パターン B: しんごさん起案 → claude-pipeline 経由で Plan をひとしに配送 → ひとしの Generator 実装 → ひとしの Reviewer + Evaluator
③ Reviewer/Evaluator は 系統別: パターン A=しんご系、パターン B 実装側=ひとし系
④ main への合法経路は「リリースボタン」(workflow_dispatch) 一本だけ

5main 昇格の仕組み (リリースボタン)

用語整理: 「リリースボタン」とは GitHub の管理画面に「ボタン」を 1 つ設置し、それを押すだけで develop → main へ反映する仕組みのこと。 技術的には GitHub Actions の workflow_dispatch という機能で実装する (本資料では「リリースボタン」と呼ぶ)。

なぜ Release PR ではなく「リリースボタン」なのか:

  • develop merge 前に Reviewer or 互いの review が完了している
  • main 化時点でレビュー対象はない (develop = main コピー)
  • PR は review 痕跡を残す仕組み、review がなければ形骸化する
  • リリースボタンのほうが速く、シンプル、ガードレールも明快 (1 経路のみ)

シーケンス図

workflow ファイル (リリースボタンの実体: promote-to-main.yml)

GitHub Actions workflow ファイルとして以下を .github/workflows/promote-to-main.yml に配置する。これが「リリースボタン」の実体。

name: 🚀 本番リリース (develop → main)  # サイドバー/タイトルに日本語表示
on:
  workflow_dispatch:
    inputs:
      reason:
        description: "本番リリースの理由 (Discord通知に表示。例: ADR-038 完走 / Meta 撮影前緊急修正)"
        required: true
        type: string

jobs:
  promote:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: develop
          fetch-depth: 0
          token: ${{ secrets.PIPELINE_PAT }}  # branch protection bypass

      - name: Fast-forward main to develop
        run: |
          git fetch origin main
          git merge-base --is-ancestor origin/main HEAD || {
            echo "::error::develop is not a fast-forward of main"
            exit 1
          }
          git push origin develop:main

      - name: Notify Discord
        if: always()
        run: |
          curl -H "Content-Type: application/json" \
            -d "{\"content\":\"$STATUS develop→main 昇格 by $ACTOR\n理由: $REASON\"}" \
            ${{ secrets.DISCORD_WEBHOOK_URL }}
        env:
          STATUS: ${{ job.status }}
          ACTOR: ${{ github.actor }}
          REASON: ${{ github.event.inputs.reason }}

6-2. pr-base-guard.yml

name: PR Base Branch Guard
on:
  pull_request:
    types: [opened, edited, reopened, synchronize]

jobs:
  reject-main-targeted:
    if: github.event.pull_request.base.ref == 'main'
    runs-on: ubuntu-latest
    steps:
      - name: Block main-targeted PRs
        run: |
          echo "::error::PRs targeting 'main' are not allowed."
          echo "Use the 'Promote develop to main' release button instead:"
          echo "  gh workflow run promote-to-main.yml -f reason='...'"
          exit 1
偽陽性ゼロ 正当な main 向け PR (= Release PR 等) が存在しない設計のため、このガードは正当な作業を一切阻害しない。

6-3. .claude/settings.json ask リスト

{
  "permissions": {
    "allow": [
      // 既存の読み取り系は維持
      "Bash(git status)", "Bash(git diff:*)", ...
    ],
    "ask": [
      // 追加: auto-mode でも必ず確認
      "Bash(ssh:*)",
      "Bash(ssh-keygen:*)",
      "Bash(ssh-add:*)",
      "Bash(scp:*)",
      "Bash(sftp:*)"
    ]
  }
}
permission リスト挙動auto-mode (--dangerously-skip-permissions)
allow即時実行 (確認なし)即時実行
未定義 (現状の SSH)ユーザー確認ダイアログ確認 skip = 即時実行 (危険)
ask (新規)ユーザー確認ダイアログ確認必須 (skip 不可)
deny即時拒否即時拒否
なぜ deny ではなく ask か ひとしさん本人が Claude Code 内で SSH を使う場面 (VPS log 確認等) もあるため、deny でブロックではなく確認を必ず出す方針。確認時に Mac/VPS 識別を判断して承認する。

§ ブランチ運用 (全員共通、原則) 
## ブランチ運用 (全員共通、原則)

### 不変ルール
- すべての変更は develop ブランチ経由
- 原則として main への直接 push を廃止 (branch protection で機械的にブロック)
- 本番反映は「リリースボタン」(promote-to-main.yml workflow_dispatch) のみ
- main 向け PR は CI で全部 FAIL (pr-base-guard.yml)

### develop への作業フロー
1. feature ブランチを develop から切る (`feature/{author}/{task}`)
2. PR を develop に向けて作成
3. 起案者系統の Reviewer エージェント (または互いに) で review
4. approve 後 squash merge

### main への昇格 (リリースボタンのみ)
- Web UI (推奨): Actions タブ → "🚀 本番リリース (develop → main)" → Run workflow ボタン → 理由入力
- CLI (上級者): `gh workflow run promote-to-main.yml -f reason="ADR-XXX 完走"`
- 内部動作: develop HEAD を main に fast-forward push
- 直後に deploy.yml が main push を検知して自動デプロイ (約 5 分で本番反映)
- 失敗時 (conflict 等): Discord 通知 → 手動対応
- ※ "Run workflow" ボタン文字は GitHub 固定 (英語)、ワークフロー名・入力欄は日本語表示

### なぜ Release PR を使わないか
- develop merge 前に review 完了済
- main 化時点でレビュー対象はない (develop = main コピー)
- PR は review 痕跡を残す仕組み、review がなければ形骸化
- リリースボタンのほうが速く、ガードレールも明快 (1 経路のみ)
§ 作業パターン A / B 
## 作業パターン

### パターン A: 単独開発
- ADR 不要
- 起案者本人が feature ブランチで実装
- claude-pipeline 不使用
- Reviewer 系統: 起案者本人 (本人が Reviewer エージェント起動 or 自己 review)
- 用途: bug fix、しんご手書き ADR、軽微な改修

### パターン B: Plan → 開発委譲
- ADR 起案必須 (基本しんごさんが Plan を書く)
- ADR PR を develop に出す → しんご系 Reviewer で review → merge
- ADR は claude-pipeline 経由でひとしに Plan として配送される
- ひとしが Generator を手動起動 (Actions タブから手動 trigger)
- Generator (Hikky-dev-Mac 上) が ADR plan に従い実装 → PR
- 私 (ひとし) 系の Reviewer エージェントで実装 review → develop merge
- 用途: しんごさんが Plan のみ作成、実装をひとしの Generator に委譲する場合
- 系統責任: ADR (Plan) 品質責任は起案者 (しんご)、実装品質責任は起動者 (ひとし)
§ Claude Code 利用時のプランニング規約 (全員共通) 
## Claude Code でプランを練る際のルール

### プランモード必須
全員 (しんご・ひとし) は Claude Code で実装や設計のプランを練る際、
必ず Plan mode に自動切替してから検討を進めること。

### なぜ
- Plan mode は実行系ツール (Edit/Write/Bash 書込み系) を封じる安全モード
- プランニング段階で意図しないファイル変更やコマンド実行を防ぐ
- ADR / 設計判断 / 影響範囲調査などは Plan mode で行うことで、
  「考えながら触る」事故を構造的に防止する

### 運用
- セッション開始時 or 設計検討タスク受領時に自動的に Plan mode に入る
- 実装に入る判断ができた段階で ExitPlanMode を使って通常モードに戻る
- Claude Code 側で自動切替の hook を整備 (settings.json の SessionStart hook 等)

### Plan mode と通常モードの境界
| 段階 | モード |
|---|---|
| プランニング (ADR ドラフト / 影響範囲調査 / 設計検討) | Plan mode |
| 確定した実装 (commit / push / 設定変更) | 通常モード |
| Reviewer エージェント起動 | 通常モード (実行は agent 側で制御) |
