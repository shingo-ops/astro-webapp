## 1. 背景：ADR-011施行後に判明した構造的問題

ADR-011の自動レビューパイプラインは意図通り動作しましたが、運用してみて以下の構造的問題が判明しました：

### 1-1. 無限ループリスク

現行の`claude-pipeline.yml`の`Run Claude Code`ステップは以下のプロンプトです：

```
"次のADRファイルを読み取り、内容のレビューと実装を行ってください: ${ADR_FILES}。
成果は feedback.md に追記してください。"
```

このプロンプトには**終了条件が一切含まれていません**。「何か見つけたら報告せよ」という指示しかなく、「問題がなければ終われ」という指示がない。結果として、毎回必ず何らかの指摘が出力され、Revision無限ループが発生する構造です。

### 1-2. 終了条件追加では解決しない（誤陰性リスク）

「Criticalが0件ならLGTMで終了」というルールを追加する案を検討しましたが、以下の理由で破棄しました：

学術論文（arXiv 2603.00539）と業界調査（CodeRabbit 470 PR分析）で確認されている通り、AIレビュアーに重大度フィルタを与えると**真のCritical問題が降格分類されて見逃される**現象が発生します。

具体例：
- マルチテナント間データ混入バグ → 「CIは通る」のでMajor扱い → 切り捨て
- Wise webhook署名検証漏れ → 「即座に壊れない」のでMajor扱い → 切り捨て
- PIIログ出力 → 「Minor書くな」ルールで省略 → 見逃し

これらはJarvis CRMで実際に致命的になる種類の問題です。

### 1-3. パートナーClaude Codeの資源が非効率

現行のフローでは、パートナーClaude Codeは以下のように使われています：

```
コードベース全体を読める環境にいる
　↓
しかし抽象的なADR文章だけを読んでレビュー文章を書く
　↓
コードベースの文脈を活かさずに終わる
```

これは資源の浪費です。実コードを触れる立場にいるなら、実コードに反映する方が価値が高い。

**Before（現行 ADR-011）:**
```

　　　　　　　　　　　　　　　　　　　　　　　　↓（いつか終わる）
　　　　　　　　　　　　　　　　　　　　　　　パートナーが実装
```

**After（本提案）:**
```
ADR起案（AC明記） → パートナーが直接実装（コードベース最適化込み） → PR
　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　↓

　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　↓
　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　Shingoマージ
```

## 3. なぜこれが優れているか

### 3-1. 誤陰性リスクの構造的排除

レビュアーに「Criticalだけ報告」と指示する必要がなくなります。実装が動けば設計は妥当、動かなければ修正する。**実装そのものが設計の検証**になるため、抽象的なレビューでは絶対に捕まえられない問題（実コードとの矛盾、既存実装との衝突等）が自動的に顕在化します。

### 3-2. レビュー誤陽性問題の回避

業界平均の偽陽性率は60〜80%（Jet Xu Engineering Blog、22,000件分析）。Shingoのケースでも実際にRev1→Rev2→Rev3とMinor指摘が積み上がる兆候がありました。実装→検証モデルにすればこの問題は構造的に発生しません。

### 3-3. パートナーClaude Codeの能力の最大活用

レビューだけでなく実装まで担当することで、コードベース全体の文脈を活かした最適化が可能になります。例：

- ADRが`docs/adr/README.md`の作成を求めている → 既存の他ADRをスキャンして自動的に正しいリンクを生成
- ADRが`migration_013`を求めている → 既存マイグレーションのパターンに合わせて実装
- ADRが新しいテーブルを求めている → 既存schemaとの整合性を確認しながら設計

これらは抽象的レビューでは生成できない価値です。

### 3-4. サイクル時間の短縮

レビュー往復（1回あたり5〜10分 × 複数回）が消えます。ADR起案 → 実装 → 検証 → マージの直線フロー。

## 4. 必要な技術変更

### 4-1. `claude-pipeline.yml` の改修

`Run Claude Code`ステップのプロンプトを以下方針に変更：

```
（旧）レビューと実装を行ってください
　↓
（新）以下のADRに基づいて実装を行ってください。
　　　- ADRに記載されたAcceptance Criteria（AC-XXX）を全て満たすこと
　　　- 既存コードベースのパターンに整合させること
　　　- 実装範囲外の変更は行わないこと
　　　- 設計意図が不明確な場合は questions/QXX.md で停止して相談すること
```

具体的なYAML実装はパートナーClaude Codeの判断に委ねます。

### 4-2. CLAUDE.md の更新

パートナーClaude Codeへの指示を「ADRレビュー担当」から「ADR実装担当」に変更。

### 4-3. `docs/operational-runbook.md` の新規作成

Web Claudeの役割を「実装diff検証担当」として明記。GitHubコネクタ設定もここに記載（ADR-011で言及済みだが未実装）。

## 5. ADR起案時の新しい必須要件

設計と実装の整合性を担保するため、今後のADRには以下を必須化します：

### 5-1. Acceptance Criteria（AC）の機械的チェック可能性

例（Before）:
> ADRと実装の整合性を保つ

例（After）:
> - AC-001: `docs/adr/`配下のADR命名規則違反ファイルが0件であること
> - AC-002: パイプラインがdevelopブランチで自動起動すること
> - AC-003: feedback.md以外の変更があるPRはauto-mergeしないこと

### 5-2. 実装範囲の境界

「何を含むか」だけでなく「何を含まないか」を明記。スコープクリープを防ぐ。

### 5-3. 既存コードベースとの予想される接点

実装時に確認すべきファイル・テーブル・API endpointを事前に列挙。
