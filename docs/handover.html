プロジェクト概要
Sales Anchor (Jarvis CRM) とは何か

技術スタック
システムを構成する技術と、それぞれの役割

このシステムは「フロント（画面）」「バック（処理）」「DB（データ保管）」「インフラ（サーバー）」が それぞれ分業して動いています。料理店に例えると分かりやすいです。

バックエンド
Python 3.12 / FastAPI / SQLAlchemy 2.0 + asyncpg
ユーザーのリクエストを受け取り、データベースからデータを取得・加工して返す「頭脳」。
レストランの厨房。注文を受け、料理を作り、皿に盛って渡す役割。
フロントエンド
React 18 + TypeScript + Vite
ブラウザに表示される画面そのもの。ユーザーが操作するボタン・フォーム・一覧表などすべて。
レストランのホール。お客さんが見て触れる空間。
データベース
PostgreSQL 16（マルチテナント・スキーマ分離）
顧客データ・メッセージ・見積・請求などを永続的に保存する。会社（テナント）ごとに棚が完全分離。
倉庫。Aさんのものと Bさんのものは別々の鍵付き棚に厳重管理。
インフラ
Docker Compose / さくらVPS / GitHub Actions
アプリが動く「土台」。Dockerで環境をパック、VPSが本番サーバー、GitHub Actionsが自動デプロイ担当。
お弁当箱（Docker）に詰めて、トラック（VPS）で配達、配送ロボ（Actions）が自動出荷。
Redis
キャッシュ / SSE Pub/Sub / レート制限
高速なメモリ上のデータ置き場。DB より 100 倍速い。リアルタイム通知（SSE）やブルートフォース対策にも使う。
厨房の手元メモ帳。毎回倉庫に走らなくても、よく使う情報はここに貼っておく。
Celery
非同期タスクキュー / バッチ定期実行
ユーザーが関係ない「裏方作業」を担当。毎朝5時に古いログを自動削除したり、重い処理を後回しで実行。
夜間清掃スタッフ。お客さんがいない時間帯に黙々と作業してくれる。
ランディングページ (LP)
Astro — salesanchor.jp
新規顧客向けの紹介ページ。React ではなく Astro という軽量フレームワークを使い、表示速度を最優先にしている。アプリ本体（app.salesanchor.jp）とは別管理。
お店の看板・ショーウィンドウ。中に入る前に外から見える部分。

04
デザインシステム
色・文字・余白を「一元管理」する仕組み（ADR-067）
このプロジェクトで最も重要な設計ルールです。 デザインの値は必ず CSS 変数（トークン）経由で参照します。 数値の直書きは自動チェックで弾かれます。

Single Source of Truth（単一の正への基準）とは？

例えば「ボタンの色を青から緑に変えたい」とき、コード 300 箇所に #3B82F6 と直書きしていたら 全部探して書き換えなければなりません。しかし 変数（トークン） を使えば、 --accent: #3B82F6 の 1 箇所だけを --accent: #10b981 に変えれば、 サイト全体が一瞬で変わります。これが「単一の正への基準」です。

ハードコードNG vs トークンOK
/* ❌ NG: ハードコード — 後で変更するとき300箇所を修正する羽目に */ .button { background: #3B82F6; border-radius: 8px; padding: 12px 16px; } .card { border-radius: 8px; opacity: 0.8; } /* ✅ OK: デザイントークン — 1変数を変えるだけで全画面に反映 */ .button { background: var(--accent); border-radius: var(--radius-md); padding: var(--space-3) var(--space-4); } .card { border-radius: var(--radius-md); opacity: var(--opacity-dim); }

自動強制システム（ADR-067 フェーズ4）— 違反はコミット不可

「ルールを人が守る」ではなく「機械が強制する」仕組みを実装しています。

ESLint チェック
TypeScript コード（TSX）内でのインラインスタイルへの数値直書き（opacity・zIndex）を 開発中リアルタイムにエラー表示。
CSS スキャンスクリプト
check-css-hardcoded-values.js が CSS ファイルを全スキャン。 border-radius・opacity・z-index の数値直書きを検出。
Pre-commit フック
git commit を実行した瞬間に自動でチェック実行。 違反があればコミット自体がブロックされる。
GitHub Actions CI
PR を出した時にも全チェックが走る。CI が赤のままではマージ不可（Branch Protection）。
ローカルで確認するコマンド
cd frontend # TSX の hex 色・inline 数値チェック npm run lint # CSS の hex 色ハードコードチェック npm run check:css-colors # CSS の数値ハードコードチェック（opacity/border-radius/z-index等） npm run check:css-values # index.css のダーク/ライト変数パリティチェック npm run check:dark-parity # 全チェック一括 npm run check:all

i18n（多言語対応）ルール

Sales Anchor は日本語・英語の切り替えに対応（react-i18next）。 UI に表示する文字列はすべて t("key") 経由にする。JSX への日本語直書きは禁止。

i18n 対応例
// ❌ NG: JSXに日本語直書き <button>保存する</button> // ✅ OK: t() 経由で参照 <button>{t("common.save")}</button> // 確認コマンド（ヒット0行であること） git diff --name-only develop...HEAD -- 'frontend/src/**/*.tsx' \ | xargs -I{} grep -nE '[ぁ-んァ-ヶ一-龯]' {} 2>/dev/null

デプロイの仕組み（deploy.yml）

main ブランチへのマージをトリガーに GitHub Actions が自動実行。 VPS に SSH 接続し、コード更新 → Docker 再起動 → DB マイグレーションの 3 ステップを順番に行う。

デプロイ 3ステップ（deploy.yml の構造）
Step 1: Deploy to VPS (code + env + docker compose) → git pull + docker compose up でコードと環境変数を更新 Step 2: Run database migrations (part 1) → migration 001〜060番台を順番に実行（IF NOT EXISTS で冪等） Step 3: Run database migrations (part 2) + finalize → migration 061〜072番台 + ヘルスチェック ※ Step1 が失敗すると Step2/3 は自動スキップ ※ 全 migration は IF NOT EXISTS で設計 → 再実行しても安全
新しい migration を追加したときは deploy.yml への記載も必須。 記載漏れは migration-guard CI が自動検知してブロックする。

INFO
VPS コンテナの注意事項
VPS の /app 配下は書き込み不可（appuser 権限）。スクリプト出力先は /tmp を使う。 /tmp は tmpfs なので docker compose cp は使えない。 ファイル取り出しは docker compose exec -T backend cat /tmp/xxx > host_file のみ。
INFO
develop → main も PR 経由（直 push 禁止）
main は Branch Protection で直 push が物理的に禁止されている。 Hikky-dev（Claude Code）が PR を作成し、しんごさんがマージする流れ。 緊急時は admin（しんごさん）のみ bypass 可。
OK
DB直接操作（開発フェーズ中）
現在は開発フェーズのため、psql や docker exec psql による直接 INSERT/UPDATE は OK。 ただし 暗号化済みシークレット（*_token_encrypted）の手動投入 と destructive 操作（DROP TABLE / TRUNCATE） は常に PO 確認必須。

# コンテナ状態確認（VPSにSSH後） docker compose ps # バックエンドログ確認 docker compose logs backend --tail=50 # 本番DBに接続（テナントコードはしんごさんに確認） docker compose exec postgres psql -U jarvis -d jarvis_db > SET search_path TO 【本番スキーマ名】; # migration を特定テナントに手動実行 gh workflow run run-migration.yml -f tenant_code=【本番テナントコード】 # ファイルをVPSから取り出す（/tmp の制約あり） docker compose exec -T backend cat /tmp/output.csv > ./local_output.csv # GitHub Actionsのデプロイ状況確認 gh run list --workflow deploy.yml --limit 5

アーキテクチャ上の重要な制約

注意
/app 配下は書き込み不可
VPS の Docker コンテナ内 /app は appuser 権限で書き込み不可。スクリプトの出力先は必ず /tmp を指定する。
注意
/tmp は tmpfs — コンテナ再起動で消える
docker compose cp backend:/tmp/xxx は使えない（Could not find the file エラー）。ホストへの取り出しは docker compose exec -T backend cat /tmp/xxx > host_file 一択。
INFO
SQLite では動作確認不可
マルチテナント RLS・Webhook・外部 API 経路の検証は必ず PostgreSQL 実機で行う。SQLite + AsgiTransport は「動作確認」として認められない（ADR-038）。
