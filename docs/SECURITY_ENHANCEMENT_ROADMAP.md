## 3. 対策①：監査役 Claude Code の常駐化（3層構成）

### 3-1. たとえ話

**警備会社と契約するのと同じ構造**を Claude Code で作ります。

| 層 | たとえ | いつ動く | コスト |
|----|-------|---------|--------|
| Tier 1: pre-commit hook | **玄関の警備員** | コミットの瞬間 | ほぼ0（差分のみ） |
| Tier 2: GitHub Actions | **24時間巡回するガードマン** | PR + 6時間ごと | 約500Kトークン/週 |
| Tier 3: subagent | **顧問契約のセキュリティ専門家** | hitoshiさんが呼んだ時 | 都度API費用 |

### 3-2. なぜ「サブエージェント単体」も「複数ターミナル」もダメか

| 案 | 問題点 |
|----|--------|
| **サブエージェント単体** | セッション内でしか動かない。親が終了したら消える。**24/7監査には不向き** |
| **複数ターミナル** | 手動で起動・管理が必要。Mac閉じたら止まる。worktreeでファイル競合は防げるが**自動化されない** |
| **Hooks に Claude を呼ばせる** | Hooks は「反応する」だけで Claude セッションを起動できない（よくある誤解） |
| **`/loop` コマンド** | セッション開いてる間だけ。7日で自動失効 |

### 3-3. アーキテクチャ図

```
┌──────────────────────────────────────────────────────┐
│ Tier 1: pre-commit hook（Mac側、コミット時）          │
│ → claude -p で差分のシークレット・SQLi をブロック     │
│ → 数秒で完了、危険なら commit を拒否                  │
└──────────────────────────────────────────────────────┘
                       │ git push
                       ▼
┌──────────────────────────────────────────────────────┐
│ Tier 2: GitHub Actions（クラウド側、24/7）            │
│ → anthropics/claude-code-action@v1                   │
│ → 毎PR + 6時間ごとの cron で自動監査                  │
│ → 結果を PR コメント / Issue として投稿               │
│ ★これがメインの「監査役エージェント」                  │
└──────────────────────────────────────────────────────┘
                       │ 発見した脆弱性
                       ▼
┌──────────────────────────────────────────────────────┐
│ Tier 3: .claude/agents/security-auditor.md           │
│ → ローカルでhitoshiさんが対話する時の専門家ロール     │
│ → "@security-auditor この PR をレビューして"          │
└──────────────────────────────────────────────────────┘
```

### 3-4. 実装ファイル雛形（着手時にコピペで使える）

**A. サブエージェント定義（Tier 3）**
```markdown
# .claude/agents/security-auditor.md
---
name: security-auditor
description: FastAPI/React/Postgres 構成の脆弱性を専門に診断
tools: Read, Grep, Glob, Bash
---
あなたは Sales Anchor 専属のセキュリティ監査人です。
重点項目:
- tenant_id 検証漏れ（マルチテナント分離破壊）
- IDOR (Insecure Direct Object Reference)
- SQL インジェクション（特に raw SQL / f-string SQL）
- JWT 改ざん・有効期限欠落
- CORS 設定の過剰な許可
- レート制限の欠如
出力フォーマット: VULN-YYYY-NNN 形式のレポート
```

**B. GitHub Actions（Tier 2 = メイン）**
```yaml
# .github/workflows/security-audit.yml
name: Security Audit by Claude
on:
  pull_request:
  schedule:
    - cron: '0 */6 * * *'  # 6時間ごと
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            security-auditor サブエージェントを使って
            backend/app/api 配下の全エンドポイントを監査せよ。
            tenant_id 検証漏れ・IDOR・SQLi を重点的に確認。
            発見した脆弱性は PR コメントで報告。
          claude_args: "--max-turns 10 --model claude-opus-4-6"
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
```

**C. pre-commit hook（Tier 1）**
```bash
# .git/hooks/pre-commit （実行権限付与: chmod +x）
#!/bin/bash
DIFF=$(git diff --cached)
RESULT=$(echo "$DIFF" | claude -p \
  "この差分にシークレット漏洩・SQLi・認証バイパスがあれば 'BLOCK: 理由' を出力。安全なら 'OK'" \
  --allowedTools Read --output-format text)

if [[ "$RESULT" == BLOCK:* ]]; then
  echo "❌ Claude Code が危険な変更を検出: $RESULT"
  exit 1
fi
```

### 3-5. コスト見積もり

| 項目 | 月額目安 |
|------|---------|
| GitHub Actions（6時間ごと + 毎PR） | API費用 約3,000〜8,000円 |
| pre-commit hook | API費用 約500円（コミット数による） |
| サブエージェント（手動呼び出し） | API費用 必要時のみ |
| **合計** | **約5,000〜10,000円/月** |

### 3-6. 実装工数

- Tier 1 (hook): **30分**
- Tier 2 (GitHub Actions): **1時間**
- Tier 3 (subagent定義): **30分**
- 動作確認・チューニング: **2時間**
- **合計: 約半日**

## 4. 対策②：動的ペネトレーションテスト内製化

### 4-1. 前提：ステージング環境の構築

**⚠️ 動的テストは本番環境では絶対に実施しない**。必ず別環境で。

```
┌──────────────────────────────────────────┐
│ ステージング環境構成                      │
│                                          │
│ - さくらVPS 1GBプラン（約700円/月）       │
│ - ドメイン: pentest.jarvis-claude.uk      │
│ - DB: 本番の匿名化コピー                  │
│ - Cloudflare/Firebase: 別プロジェクト     │
│ - WAF例外設定（テスト中は除外）            │
└──────────────────────────────────────────┘
```

**必要文書（実装時に作成）**:
1. しんごさんへの事前承認書（自社サーバーへの攻撃許可書）
2. 緊急時の連絡先・ロールバック手順
3. テスト実施記録（日時・対象・実施者・結果）

### 4-2. 動的テストの種類とツール

| 手法 | 概要 | ツール例 | 実施頻度 |
|------|------|---------|---------|
| ポートスキャン | 開いているポートと稼働サービスを列挙 | nmap, masscan | 月1回 |
| Webスキャン | XSS, SQLi, パストラバーサルを総当たり | OWASP ZAP, Nuclei | 月1回 |
| ファジング | 異常な入力を大量送信して落とす | ffuf, wfuzz | 四半期1回 |
| 認証攻撃 | パスワード総当たり、トークン推測 | Hydra | 四半期1回 |
| API攻撃 | OpenAPI仕様書から自動攻撃 | Schemathesis | 月1回 |

### 4-3. Mac側で実行するコマンド例（着手時にコピペで使える）

```bash
# OWASP ZAP の自動スキャン
docker run --rm -v $(pwd)/zap-report:/zap/wrk \
  zaproxy/zap-stable zap-baseline.py \
  -t https://pentest.jarvis-claude.uk \
  -r report.html

# Nuclei による既知脆弱性スキャン
docker run --rm projectdiscovery/nuclei \
  -u https://pentest.jarvis-claude.uk

# nmap によるポートスキャン（ステージングVPSのIPに対して）
nmap -A -p- <ステージングVPSのIP>
```

### 4-4. Claude Code の役割

- ZAP/Nuclei のレポート（しばしば数百件の警告）を読み込み、**真の脆弱性 vs 誤検知**を判定
- HTTPリクエスト/レスポンスから影響度を評価
- 必要に応じて curl/Python で追加検証スクリプトを生成

### 4-5. コスト・工数

- **ステージング環境**: 約700円/月
- **初期構築**: 5日（環境構築 + 初回スキャン + チューニング）
- **以降**: 月1日（定期スキャン + 結果レビュー）

## 5. 対策③：ロジック攻撃テスト（Claude Code が最も得意）

### 5-1. たとえ話

> **「鍵は壊れていないが、宅配業者を装えば家族が開けてくれる」型の攻撃**

通常のスキャナーでは見つからない、**ビジネスロジックの欠陥**を狙う攻撃。Claude Code はソースコード全体を読めるので、**機械的な全数チェック**が可能。

### 5-2. 対象とする攻撃パターン

| 攻撃 | Sales Anchor での具体例 |
|------|---------------------|
| **IDOR** | `/api/customers/123` を `/api/customers/124` に変えて他テナントのデータが見える |
| **テナント分離破壊** | tenant_id を改ざんして他社のデータにアクセス |
| **権限昇格** | role=user で role=admin の API を叩く |
| **ビジネスロジック欠陥** | 注文金額を負数にして残高プラス、数量を整数オーバーフロー |
| **競合状態** | 同時に2つのリクエストを送って二重決済 |

### 5-3. Claude Code の活用フロー

```
Step 1: 「FastAPI 全エンドポイントを列挙、tenant_id 検証の有無を表に」と依頼
   ↓
Step 2: 検証漏れエンドポイントを発見
   ↓
Step 3: ステージング環境で実際に攻撃して再現（PoC作成）
   ↓
Step 4: 修正パッチ作成 → PR提出
   ↓
Step 5: 同じパターンを他のエンドポイントでも探す（横展開）
```

### 5-4. 工数

- **初回チェック（全エンドポイント走査）**: 1日
- **発見した脆弱性の修正**: 件数次第（1件あたり1〜4時間）
- **以降の継続実施**: 新APIを追加するたびに該当ファイル単位で実施

sudo ssh-audit localhost

# VPS側: Docker設定監査
docker run --rm --net host --pid host --userns host --cap-add audit_control \
  -v /var/lib:/var/lib -v /var/run/docker.sock:/var/run/docker.sock \
  docker/docker-bench-security

## 7. 対策⑤：外部業者による第三者監査（年次）

### 7-1. なぜ内製で代替できないか

| 観点 | 理由 |
|------|------|
| **第三者性** | ISMS/SOC2/Pマーク取得時、独立監査が要件 |
| **最新APT対応** | ゼロデイ・APT手法は専門業者が最新知見を持つ |
| **顧客提示エビデンス** | B2B顧客から監査レポート要求があった際、社内文書では弱い |
| **盲点の発見** | 設計者自身では「自分の知っている穴」しか探さない |

### 7-2. タイミングとコスト

| タイミング | 内容 | 予算目安 |
|-----------|------|---------|
| 本番リリース直後 | 初回フル監査 | 50〜100万円 |
| 大型機能追加時 | 差分監査 | 20〜50万円 |
| 年次定期 | 定期再監査 | 30〜80万円 |

### 7-3. 業者選定のポイント

- IPA「情報セキュリティサービス基準」適合事業者
- WASEC（OWASP公認）またはGIAC GPEN等の有資格者在籍
- レポート例の事前提示
- ステージング環境前提で受託可能か

## 8. やってはいけないこと（全対策共通）

### 🚫 絶対NG

1. **本番環境への動的スキャン**（DBが壊れる・サービス停止）
2. **Cloudflare/Firebase/さくらVPS の管理画面への攻撃**（規約違反・アカウント停止）
3. **顧客データへのアクセス試行**（個人情報保護法違反の可能性）
4. **DDoS的な高負荷スキャン**（IPブロック・近隣サーバーへ迷惑）
5. **テストツールを本番VPSにインストール**（Metasploit等は痕跡が残る）

### ⚠️ 事前確認すべき

- さくらVPSの利用規約「第三者への攻撃禁止」項目（自社サーバーは原則OK）
- Cloudflareの「Vulnerability Disclosure Policy」（CDN経由のテストは通知が必要な場合あり）
- Firebase「Acceptable Use Policy」（認証ブルートフォースは検知される）
- 個人情報保護法（テストデータは必ず匿名化・架空データを使用）
